Create `cli/lib/drift_detector.py` implementing anchor-level semantic drift detection per D-05 and D-12.

1. Define `@dataclass(frozen=True) class DriftResult` with fields:
   - `anchor_id: str`
   - `frz_ref: str`
   - `has_drift: bool`
   - `drift_type: str` — one of: `"none"`, `"missing"`, `"tampered"`, `"new_field"`, `"constraint_violation"`, `"unknown_expired"`
   - `detail: str`

2. Implement `check_drift(anchor_id: str, frz_package: FRZPackage, target_data: dict[str, Any]) -> DriftResult`:
   - **target_data contract**: a dict keyed by anchor_id (e.g., `{"JRN-001": {...}, "ENT-001": {...}}`), where each value is the extracted semantic content for that anchor. The lookup key is `target_data.get(anchor_id)`.
   - Extract the anchor's semantic content from FRZ by scanning all MSC dimensions:
     - core_journeys: match by `id` field (JRN-xxx)
     - domain_model: match by `id` field (ENT-xxx)
     - state_machine: match by `id` field (SM-xxx)
     - acceptance_contract: scan `expected_outcomes` for FC-xxx references
     - known_unknowns: match by `id` field (UNK-xxx)
   - If anchor not found in FRZ → `DriftResult(drift_type="missing", has_drift=True, detail="Anchor {id} not found in FRZ baseline")`
   - If anchor found but not in target_data → `DriftResult(drift_type="missing", has_drift=True, detail="Anchor {id} not present in extracted output")`
   - If anchor found in both, compare structural semantics:
     - Extract name/title field from both, normalize to lowercase, compare
     - If core fields differ → `DriftResult(drift_type="tampered", has_drift=True, detail="Anchor {id} semantics differ from FRZ baseline")`
     - If they match → `DriftResult(drift_type="none", has_drift=False, detail="OK")`

3. Implement `check_derived_allowed(frz_package: FRZPackage, output_data: dict[str, Any]) -> list[str]`:
   - Get `allowed = set(frz_package.derived_allowed)`
   - Define intrinsic keys always allowed: `{"artifact_type", "schema_version", "source_refs", "frz_ref", "traceability"}`
   - Return list of keys in output_data that are NOT in allowed AND NOT in intrinsic keys
   - Per D-04: field whitelist approach — only derived_allowed fields can be added

4. Implement `check_constraints(frz_package: FRZPackage, output_data: dict[str, Any]) -> list[str]`:
   - For each constraint string in `frz_package.constraints`, check if it appears as a key in output_data with a value that contradicts it
   - Simplified: check if constraint text (or key portion before colon) is absent from output_data's keys
   - Return list of violated constraints

5. Implement `check_known_unknowns(frz_package: FRZPackage, output_data: dict[str, Any]) -> list[dict[str, str]]`:
   - For each KnownUnknown in `frz_package.known_unknowns`:
     - If `ku.status == "open"` and `ku.expires_in` indicates expiry (contains "0" or "expired")
     - Return `{"id": ku.id, "topic": ku.topic, "status": "expired"}`

6. All functions use `CommandError` + `ensure()` for preconditions.
7. All function signatures have type annotations. Use `from __future__ import annotations`.
8. Import from `cli.lib.errors`, `cli.lib.frz_schema`, `cli.lib.anchor_registry`.
9. Do NOT create CLI entry point — this is a pure library module called by extract modes.
  </action>

    - check_derived_allowed returns ["tech_details"] when derived_allowed=["ui_layout"] and output has tech_details
    - check_constraints returns violated constraint strings when output lacks required keys
