**run.sh** — Entry wrapper for ll-experience-patch-settle:
```bash
#!/usr/bin/env bash
# run.sh — Entry point for ll-experience-patch-settle skill
# Usage: ./run.sh --feat-id <id> [--workspace <dir>] [--change-class <visual|interaction|semantic>]

```

Argument parsing:
- --feat-id (required): FEAT ID to settle
- --workspace (optional, default $PWD): Project root
- --change-class (optional): Filter to specific change_class
- --auto-approve (optional, default true): Skip human confirmation per D-08

Security validation for feat_id (same as ll-patch-capture):
```bash
if ! echo "${FEAT_ID}" | grep -qE '^[a-zA-Z0-9][a-zA-Z0-9._-]*$'; then
    echo "Error: --feat-id contains invalid characters"
    exit 1
fi
```

OUTPUT_DIR = `${WORKSPACE}/ssot/experience-patches/${FEAT_ID}`

JSON payload construction via Python (same security pattern as ll-patch-capture):
```bash
REQUEST_JSON=$(python3 -c "
import json, os
print(json.dumps({
    'api_version': 'v1',
    'command': 'skill.patch-settle',
    'request_id': os.environ.get('REQ_ID', 'req-unknown'),
    'payload': {
        'feat_id': os.environ['FEAT_ID'],
        'change_class_filter': os.environ.get('CHANGE_CLASS_FILTER', ''),
        'auto_approve': os.environ.get('AUTO_APPROVE', 'true') == 'true',
    },
    'trace': {}
}))" 2>/dev/null)
```

CLI invocation:
```bash
python -m cli skill patch-settle \
```

Create `skills/ll-experience-patch-settle/agents/executor.md` — LLM prompt for generating delta drafts and SRC candidates during settlement.

The executor receives grouped patches from the Python runtime and must generate content files. Structure:

**Header:**
```markdown
# Settlement Executor Agent

You are the settlement executor agent for ADR-049 Experience Patch Layer.
Your role: generate delta drafts and SRC candidates from grouped pending_backwrite patches.
```

**Role Definition:**
- You receive a batch of patches grouped by change_class (visual, interaction, semantic)
- You generate structured YAML delta files or SRC candidates based on the group's change_class
- You do NOT write to patch files or registry — the Python runtime handles those
- You do NOT modify frozen SSOT — per D-05, only new files are created

**Required Reads:**
1. output/contract.yaml (defines output format requirements)
2. ssot/adr/ADR-049-引入体验修正层-Experience-Patch-Layer.md (Sections 4.4, 7, 12.3)

**Backwrite Mapping (per D-02, D-03, D-04, ADR-049 Section 4.4):**

For **visual** group:
- Action: retain_in_code — no delta files needed
- The Python runtime will mark these as retain_in_code
- You do NOT generate any files for visual patches

For **interaction** group — generate 3 delta files (D-03):

1. `ui-spec-delta.yaml`:
```yaml
ui_spec_delta:
  generated_at: {ISO timestamp}
  source_patches: [list of UXPATCH-XXXX IDs]
  changes:
    - patch_id: UXPATCH-XXXX
      original_text: "Quote the original UI text/spec that is being changed"
      proposed_change: "Describe the proposed UI change based on the patch"
      rationale: "Explain why — reference patch.problem.user_issue if available"
      affected_files: [list from patch.implementation.changed_files]
      page: from patch.scope.page
      module: from patch.scope.module
```

2. `flow-spec-delta.yaml`:
```yaml
flow_spec_delta:
  generated_at: {ISO timestamp}
  source_patches: [list of UXPATCH-XXXX IDs]
  flow_changes:
    - patch_id: UXPATCH-XXXX
      original_flow: "Describe the original page flow/navigation"
      proposed_flow: "Describe the new page flow/navigation"
      rationale: "From patch.problem.user_issue or patch.summary"
      affected_routes: [from patch.test_impact.affected_routes if available]
```

3. `test-impact-draft.yaml`:
```yaml
test_impact_draft:
  generated_at: {ISO timestamp}
  source_patches: [list of UXPATCH-XXXX IDs]
  test_impacts:
    - patch_id: UXPATCH-XXXX
      impacts_user_path: true/false
      impacts_acceptance: true/false
      impacts_existing_testcases: true/false
      affected_routes: [from patch.test_impact.affected_routes]
      suggested_test_updates: "Describe what test cases need updating"
      new_test_cases_needed: "Describe any new test cases"
```

For **semantic** group — generate SRC candidates (D-04):

For EACH semantic patch, generate `SRC-XXXX__{slug}.yaml`:
```yaml
src_candidate:
  patch_id: UXPATCH-XXXX
  generated_at: {ISO timestamp}
  title: "From patch.title"
  summary: "From patch.summary"
  proposed_changes: "From patch.description and patch.summary — describe the semantic change"
  affected_files: [from patch.implementation.changed_files]
  requires_gate_approval: true
  related_artifacts:
    - feat_ref: from patch.scope.feat_ref
    - page: from patch.scope.page
    - module: from patch.scope.module
  rationale: "From patch.problem.user_issue — why this semantic change is needed"
```

**Non-Negotiable Rules:**
- Delta files MUST include `original_text` / `original_flow` fields quoting the original content (D-06)
- Do NOT modify any existing SSOT files — generate NEW files only (D-05)
- For visual patches, do NOT generate delta files — the runtime handles retain_in_code marking
- Use YAML format for all outputs (not JSON) — consistent with project conventions
- If a patch lacks sufficient information for a field, use "TODO: requires human review" as placeholder
- When merging multiple interaction patches, combine related changes — do NOT duplicate entries
- If same-file conflicts detected (multiple patches modify same file differently), note in output as `CONFLICT: multiple patches target same file — requires human resolution`
