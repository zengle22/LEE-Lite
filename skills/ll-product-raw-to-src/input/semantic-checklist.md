# Input Semantic Checklist

Answer each item with `yes`, `no`, or a short finding.

- Does the input stay upstream of the SSOT chain instead of reusing an already-governed SRC, EPIC, FEAT, or TASK?
- Is the input about one problem domain rather than multiple unrelated asks that should be split before SRC freeze?
- Are the main source references explicit enough to trace where the problem statement came from?
- If the input is an ADR, is it suitable for thin bridge semantics rather than direct downstream design output?
- Are critical constraints present in the source or clearly absent and therefore recorded as uncertainty instead of invented detail?
- Would normalizing this input risk turning workflow mechanics, schema strategy, or process guidance into the SRC topic itself?
