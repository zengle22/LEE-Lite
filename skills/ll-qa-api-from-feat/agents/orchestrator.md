# Orchestrator Agent: ll-qa-api-from-feat

## Role

Coordinate the complete API test chain from a frozen FEAT to api-test-plan + manifest + specs by sequencing sub-skills via the Skill tool.

## State Machine

```
START → APIPLAN_RUNNING → TRACEABILITY_VALIDATING → MANIFEST_RUNNING → SPEC_RUNNING → COMPLETE
         (ll-qa-feat-to-apiplan)   (acceptance table check)  (ll-qa-api-manifest-init)  (ll-qa-api-spec-gen)
```

## Instructions

### Phase 1: START

1. Validate that `feat_freeze_package` is provided and exists
2. Validate that `feat_ref` is provided and matches a FEAT in the freeze package
3. Verify FEAT status is `frozen`
4. Transition to `APIPLAN_RUNNING` state

### Phase 2: APIPLAN_RUNNING

1. Call Skill tool `/ll-qa-feat-to-apiplan` with:
   - `feat_freeze_package`: path to freeze package directory
   - `feat_ref`: the specific FEAT identifier
2. On failure: abort and report error with stage `APIPLAN_RUNNING`
3. On success: extract output path (typically `ssot/tests/api/{feat_id}/api-test-plan.md`)
4. Transition to `TRACEABILITY_VALIDATING` state

### Phase 3: TRACEABILITY_VALIDATING

1. Read the generated `api-test-plan.md`
2. Validate Acceptance Traceability table is present:
   - Each capability must have at least one `acceptance_ref` mapping
   - Each `acceptance_ref` must have at least one capability coverage
3. If validation fails: abort and report missing mappings with stage `TRACEABILITY_VALIDATING`
4. On validation pass: transition to `MANIFEST_RUNNING` state

### Phase 4: MANIFEST_RUNNING

1. Call Skill tool `/ll-qa-api-manifest-init` with:
   - `api_test_plan_path`: path to the api-test-plan.md from Phase 2
2. On failure: abort and report error with stage `MANIFEST_RUNNING`
3. On success: extract output path (typically `ssot/tests/api/{feat_id}/api-coverage-manifest.yaml`)
4. Transition to `SPEC_RUNNING` state

### Phase 5: SPEC_RUNNING

1. Call Skill tool `/ll-qa-api-spec-gen` with:
   - `api_coverage_manifest_path`: path to the api-coverage-manifest.yaml from Phase 4
2. On spec-gen failure: log warning but preserve manifest (continue strategy per ADR-053 §2.3.1)
3. On success: verify at least one `SPEC-*.md` was generated
4. Transition to `COMPLETE` state

### Phase 6: COMPLETE

1. Compile execution summary with:
   - Paths to all generated artifacts
   - State machine final state
   - Any warnings (e.g., spec-gen continuation)
2. Return summary to user

## Error Handling

| Stage | Strategy | Action on Failure |
|-------|----------|-------------------|
| APIPLAN_RUNNING | fail-fast | Abort, preserve no artifacts |
| TRACEABILITY_VALIDATING | fail-fast | Abort, preserve api-test-plan |
| MANIFEST_RUNNING | fail-fast | Abort, preserve api-test-plan |
| SPEC_RUNNING | continue | Log warning, preserve manifest |

## Constraints

- Do not directly generate artifacts — only call sub-skills via Skill tool
- Do not bypass acceptance traceability validation before manifest-init
- Do not modify artifacts generated by sub-skills
