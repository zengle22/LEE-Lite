# ADR-001：LL 采用双会话双队列闭环模型作为 v0.1 执行架构

* 状态：Accepted
* 日期：2026-03-20
* 决策者：Le Zeng
* 适用范围：LEE Lite（LL）v0.x
* 相关 ADR：无

---

# 1. 背景

LL（LEE Lite）的目标，不是立即构建一个完整的 AI 工程治理平台，而是在当前阶段以最小工程量，先把需求链路稳定跑通。

当前最核心的目标是：

* 将 `raw -> src -> epic -> feat -> task` 这条链路跑稳。
* 保留关键节点的人类治理能力，而不是将 gate 交给 LLM 自判。
* 保留长程运行可能，但不再投入建设重型调度器、状态机、任务数据库等平台能力。
* 将复杂度优先压到 skill、contract、validation、evidence、artifact 上，而不是 runtime 平台本身。

此前的讨论形成了以下共识：

1. 每个 workflow 应包装为标准 skill。
2. skill 应分层：

   * L3：完整闭环 workflow 单元；
   * L2：串联多个 L3 的阶段流水线；
   * L1：串联多个 L2 的长程交付流。
3. human gate 不应由 skill 内部或 LLM 自判通过，而应外置为可审计的 gate request / decision 文件。
4. runtime 临时态不应进入工程主目录，工程目录只保留可复用、可审阅、可版本化、可追溯的正式产物。
5. 不应再开发独立 LL 调度器，以避免平台工程继续膨胀。

基于以上背景，需要为 LL v0.1 确定一套正式的执行架构决策。

---

# 2. 问题陈述

LL 需要解决的核心问题不是“如何设计一个最强的调度平台”，而是：

* 如何在不引入重型 runtime 的前提下，实现 skill 链式推进；
* 如何在关键节点可靠停住，等待 human gate；
* 如何在 gate 通过后继续推进后续 skill；
* 如何在工程目录内保持状态可见、产物可追溯、边界可审计；
* 如何让系统具备最小长程运行能力，而不是依赖人工手动接力。

如果继续建设独立调度器、状态机、事件总线或数据库驱动的 runtime，工程量会迅速膨胀，并重新成为当前阶段的主矛盾。

因此需要一个更轻、更稳、更接近当前目标的架构方案。

---

# 3. 决策

## 3.1 决策摘要

LL v0.1 采用：

## **双会话双队列闭环模型**

作为正式执行架构。

该模型以 **skill-first** 为核心，通过 **文件化 job / gate / decision / artifact / evidence / report** 在工程目录中交换状态信息，由两个长期运行的 agent 会话分别消费自动执行队列与人工待处理队列，在不引入独立调度器的前提下，实现 skill 链式推进与 human gate 闭环。

---

## 3.2 正式定义

LL v0.1 的执行架构定义如下：

1. **Skill 是执行单元**

   * L3 skill 是完整闭环 workflow 单元。
   * L2 skill 串联多个 L3 skill，形成阶段流水线。
   * L1 skill 串联多个 L2 skill，形成更长程交付流。

2. **文件是消息与状态载体**

   * job 文件用于推进下一步 skill；
   * gate request / decision 文件用于管理 human gate；
   * artifact、evidence、report、lineage 文件用于保存正式真相与审计结果。

3. **双会话分工运行**

   * 会话 A：Execution Loop，负责消费自动执行队列并触发 skill；
   * 会话 B：Human Gate Loop，负责监控待人工处理队列并提示人类处理 gate。

4. **双队列驱动闭环**

   * Execution Queue 负责自动推进；
   * Human Queue 负责人类介入与 gate 回流。

5. **不开发独立 LL 调度器**

   * 不构建独立 orchestration runtime、任务数据库、复杂状态机、事件总线。
   * 长程运行通过 job、gate、decision 文件与双 loop 实现。

6. **Gate 外置**

   * human gate 不在 skill 内等待，不由 LLM 自判通过；
   * skill 到达 gate 点时正常结束，并写入 gate request；
   * 后续是否继续由 decision 文件决定。

7. **Runtime 不进入工程主树**

   * 工程目录只保留 runtime outcome，不保留 runtime 临时态。

# 4. 架构说明

## 4.1 会话职责

### 会话 A：Execution Loop

职责：

* 扫描自动执行队列中的 ready job；
* claim 一个 job；
* 调用对应 skill；
* 生成 artifact / evidence / report；
* 如需人工审批，则生成 gate request 与 waiting-human 项；
* 如可自动继续，则生成下一个 ready job；
* 更新 job 状态为 done 或 failed。

约束：

* 每次只处理一个 job；
* 不做语义裁决；
* 不做人类审批；
* 不做复杂全局优化；
* 不在 skill 内递归调回执行器。

### 会话 B：Human Gate Loop

职责：

* 扫描等待人工处理队列与 gate request；
* 发现新的待审批项；
* 提示人类处理；
* 人工完成后写入 decision 文件；
* 更新 human queue 状态。

约束：

* 不执行业务 skill；
* 不替代人类审批；
* 不修改业务 artifact；
* 不推进自动执行队列。

---

## 4.2 双队列定义

### Execution Queue

用于自动推进 skill：

* `ready/`
* `running/`
* `done/`
* `failed/`

### Human Queue

用于人工处理与 gate 闭环：

* `waiting-human/`
* `approved/`
* `rejected/`

---

## 4.3 文件角色分离

LL v0.1 强制以下分离：

### Artifact

业务真相与正式产物。

### Job

流程推进指令，不承载业务正文。

### Gate Request

需要人工审批的请求。

### Gate Decision

人工审批结果。

### Evidence

执行与监督证据。

### Report

validation / review / freeze / repair / run 等正式报告。

### Lineage

产物追溯链。

---

# 5. 关键边界与 Invariants

LL v0.1 明确以下硬约束：

## 5.1 Skill 边界

1. skill 不自带执行器。
2. 谁执行 skill，谁就是执行器。
3. skill 内只允许调用 tooling CLI。
4. skill 内禁止调用 control CLI 触发新的执行循环。
5. skill 到达 gate 点时必须正常结束，而不是暂停等待。

## 5.2 CLI 边界

### Tooling CLI

允许 skill 调用，例如：

* `ll validate ...`
* `ll artifact ...`
* `ll evidence ...`
* `ll gate create`
* `ll gate verify`

### Control CLI

仅允许 loop 或外部控制层调用，例如：

* `ll queue claim`
* `ll queue move`
* `ll trigger job`
* `ll loop status`
* `ll resume`

## 5.3 Directory 边界

1. `ll-spec/` 是规则、schema、contract 的单点定义层。
2. `skills/` 只放 workflow skill 定义。
3. `artifacts/` 只放正式产物、审计结果、流程文件。
4. runtime 临时态不进入工程主树。

## 5.4 Gate 边界

1. human gate 外置。
2. gate 通过必须依赖 decision 文件，而不是自然语言确认。
3. LLM 不允许自判 human gate 通过。

## 5.5 Queue 边界

1. job 必须有唯一 ID。
2. job 必须显式状态迁移。
3. loop 只消费结构化 ready item。
4. 每个 loop 当前阶段仅允许单消费者，不做并发抢占。

# 7. 技术取舍

## 7.1 选择该方案的原因

### 1. 最符合当前阶段目标

当前目标是尽快把需求链跑稳，而不是继续开发平台。

### 2. 工程量最小

相较于独立调度器、状态机、事件总线，此方案实现成本最低。

### 3. 保留治理能力

通过 gate、evidence、report、contract 仍然能保持治理与审计能力。

### 4. 保留长程运行空间

通过 L2/L1 分层与双 loop，可逐步扩展为更长链路运行。

### 5. 调试与恢复更直接

文件状态清晰，便于查看当前运行位置、失败原因与审批状态。

---

## 7.2 明确不选的方案

### 方案 A：独立 LL 调度器

不选原因：

* 工程量过大；
* 当前不是主矛盾；
* 极易重新膨胀成平台工程。

### 方案 B：单会话一路跑到底

不选原因：

* 会话不稳定；
* gate 不可靠；
* 不利于恢复与审计。

### 方案 C：纯 skill 串联，不做 job/gate/queue

不选原因：

* 短链可行，长链易断；
* human gate 难以优雅接入；
* 不适合后续长程运行。

---

# 8. 风险与不足

## 8.1 当前方案的主要风险

### 1. Recovery Loop 尚不完整

当前 continuation loop 已基本形成，但失败后的 repair / revalidate / requeue 规则仍需补强。

### 2. Anti-Entropy 机制偏弱

若不尽快定义 golden principles 与 cleanup/drift 机制，系统会逐步出现规则漂移、skill 膨胀、report/evidence 格式漂移、dead job 累积等问题。

### 3. 动态 Legibility 还不够强

当前流程状态可见性较强，但 build/test/run/UI 层的运行可见性仍偏弱。

### 4. 吞吐能力有限

该方案偏单消费者、单链路稳定推进，不适合高并发吞吐场景。

# 9. 实施优先级

## 9.1 P0：当前必须完成

### 决策冻结

* 冻结本 ADR；
* 冻结双会话双队列闭环模型；
* 冻结不开发独立调度器的决策。

### 基础规则

* 目录规则定稿；
* invariants 集中定义；
* tooling CLI / control CLI 边界定稿；
* job / gate / decision 文件格式定稿。

### 最小闭环

跑通：

* `raw-to-src`
* `src-to-epic`
* `epic-to-feat`

并至少覆盖 2 个 human gate。

### Recovery 协议初版

明确：

* 自动 repair 是否允许；
* 最大 repair 轮数；
* repair 成功后的 revalidate；
* repair 失败后的处理路径；
* requeue 规则。

## 9.2 P1：短期补强

### 运行可见性

补齐：

* build report
* test report
* run report
* error report

### 反熵最小集

补齐：

* golden principles
* drift report
* cleanup checklist 或 cleanup skill

### L2 模板

定义标准 L2 pipeline 结构，串联多个 L3。

## 9.3 P2：中期扩展

* 多 worker / 并发消费；
* 更强 watcher/daemon；
* Agent SDK 驱动执行器；
* 外部审批系统接入；
* 更强 pipeline DSL。

# 11. 成功判定标准

LL v0.1 达成的标志不是“平台功能很多”，而是：

1. 能稳定从 `raw` 跑到 `feat`；
2. 在 gate 处能可靠停住；
3. 人类处理后能自动恢复推进；
4. 失败能进入标准化恢复路径；
5. 目录语义不乱，状态与产物可追溯。

---

# 12. 结果与后果

## 12.1 正面结果

* 大幅降低平台工程量；
* 更快验证 skill-first 路线；
* 保留 human gate 与治理能力；
* 为后续 L2/L1 扩展留出路径；
* 使状态、消息、审计都显式文件化。

## 12.2 负面结果

* 当前吞吐量有限；
* 并发能力弱；
* 对文件规范依赖强；
* 中长期必须补 anti-entropy，否则会逐步腐化。

# 14. 最终结论

LL v0.1 正式采用 **双会话双队列闭环模型** 作为执行架构。

这是当前阶段的最优工程取舍：

* 用 skill 代替重型 workflow runtime；
* 用文件化 job/gate/decision 代替独立调度系统；
* 用双 loop 代替复杂 orchestrator；
* 用 artifact/evidence/report 保持治理与审计；
* 在不牺牲 human gate 的前提下，获得最小长程运行能力。

本 ADR 通过后，LL 后续工作应以“先稳闭环、再补恢复、再控熵、最后扩长链”为路线，不再重新进入独立调度器建设。
