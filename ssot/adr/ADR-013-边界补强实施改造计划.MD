# ADR-013 边界补强实施改造计划

* 状态：Proposed
* 日期：2026-04-03
* 目标 ADR：[ADR-013-FEAT-to-TECH Lite-Native Skill 设计派生冻结基线](E:\ai\LEE-Lite-skill-first\ssot\adr\ADR-013-FEAT-to-TECH Lite-Native Skill 设计派生冻结基线.MD)
* 范围：`ll-dev-feat-to-tech`、`ll-dev-tech-to-impl`、相关样例与验证链

---

# 1. 改造目标

本计划用于把 ADR-013 修订后的新增治理要求落到现有仓库实现上，重点覆盖五类变化：

1. 把正式输入升级为 `feat_freeze_package + feat_ref + integration_context`。
2. 把 need assessment 从两项升级为四项：`arch_required`、`api_required`、`integration_context_sufficient`、`stateful_design_present`。
3. 新增 `integration_sufficiency_check`，并把它放到 freeze-ready 之前。
4. 把 `TECH` 的正式结构从六块升级为十块。
5. 扩展 `handoff-to-tech-impl.json` 与下游 `ll-dev-tech-to-impl` 的消费契约，防止 `IMPL` 重新猜测边界。

---

# 2. 现状缺口

## 2.1 输入 contract 还没有 `integration_context`

当前文件：

* `skills/ll-dev-feat-to-tech/input/contract.yaml`
* `skills/ll-dev-feat-to-tech/input/schema.json`
* `skills/ll-dev-feat-to-tech/scripts/feat_to_tech_common.py`

当前现状：

* 只校验 `feat_freeze_package`
* 只要求 `feat_ref`
* `validate_input_package()` 没有对 `integration_context` 做存在性、结构性和语义充分性校验

## 2.2 need assessment 仍只有 `arch_required / api_required`

当前文件：

* `skills/ll-dev-feat-to-tech/output/schema.json`
* `skills/ll-dev-feat-to-tech/output/template.md`
* `skills/ll-dev-feat-to-tech/scripts/feat_to_tech_derivation.py`

当前现状：

* `assess_optional_artifacts()` 只给出 `arch_required` 和 `api_required`
* 没有 `integration_context_sufficient`
* 没有 `stateful_design_present`
* 没有相应 rationale 和 package-level gate 绑定

## 2.3 当前 consistency gate 无法拦截“集成信息不够但仍 freeze”

当前文件：

* `skills/ll-dev-feat-to-tech/scripts/feat_to_tech_derivation.py`
* `skills/ll-dev-feat-to-tech/agents/executor.md`
* `skills/ll-dev-feat-to-tech/agents/supervisor.md`

当前现状：

* 只有 `design_consistency_check`
* 没有 `integration_sufficiency_check`
* supervisor 还没有把集成上下文充分性和 stateful design 显式性纳入通过条件

## 2.4 `TECH` 输出结构只覆盖旧六块

当前文件：

* `skills/ll-dev-feat-to-tech/output/template.md`
* `skills/ll-dev-feat-to-tech/scripts/feat_to_tech_package_builder.py`
* `skills/ll-dev-feat-to-tech/scripts/feat_to_tech_documents.py`

当前现状：

* 已有 `state_model`
* 但没有明确冻结：
  * 关键算法 / 判定逻辑
  * 输入输出矩阵 / side effects
  * 技术名词表 / canonical ownership
  * migration / compatibility 约束引用

## 2.5 下游 handoff 和 `tech_to_impl` 输入契约仍偏薄

当前文件：

* `skills/ll-dev-feat-to-tech/scripts/feat_to_tech_package_builder.py`
* `skills/ll-dev-tech-to-impl/input/contract.yaml`
* `skills/ll-dev-tech-to-impl/input/schema.json`
* `skills/ll-dev-tech-to-impl/scripts/tech_to_impl_validation.py`

当前现状：

* `handoff-to-tech-impl.json` 只保留 `feat_ref / tech_ref / arch_ref / api_ref / target_workflow / artifact refs`
* 下游只验证基础 lineage
* 没有 `integration_context_ref`、`state_machine_ref`、`canonical_owner_refs`、`migration_constraints_ref`、`algorithm_constraint_refs`

---

# 3. 实施原则

1. 先改 contract 和 schema，再改 runtime 和文档输出，最后改样例与下游消费。
2. 不推翻当前 `TECH mandatory + ARCH/API optional + unified workflow` 主链。
3. 新增字段优先走“显式缺失即 fail”的治理路径，不用隐式默认值掩盖输入不足。
4. 对 `ll-dev-tech-to-impl` 的改造必须与上游 handoff 扩展同步交付，避免出现上游先写新字段、下游仍按旧契约消费的半升级状态。

---

# 4. 分阶段改造

## Phase 1：输入契约与对象模型升级

### 目标

把 `integration_context` 引入为正式输入，并定义最小结构。

### 修改文件

* `skills/ll-dev-feat-to-tech/SKILL.md`
* `skills/ll-dev-feat-to-tech/ll.contract.yaml`
* `skills/ll-dev-feat-to-tech/input/contract.yaml`
* `skills/ll-dev-feat-to-tech/input/schema.json`
* `skills/ll-dev-feat-to-tech/resources/examples/input.example.md`

### 关键改造

* 在 skill 描述和 contract 中把正式输入改为 `feat_freeze_package + feat_ref + integration_context`
* 在 input schema 中新增 `integration_context` 节点或其等价引用结构
* 明确 `integration_context` 的最小字段：
  * workflow / loop / skill / CLI inventory
  * module boundary inventory
  * legacy fields / states / interfaces
  * canonical ownership snapshot
  * compatibility / migration constraints
  * legacy invariants
  * related refs
* 更新 input example，使样例包能表达该对象

### 完成标准

* 没有 `integration_context` 的输入不能再被视为完整正式输入
* schema 能区分“完全缺失”和“字段存在但内容不足”

## Phase 2：输入校验、assessment 与 gate 升级

### 目标

把新增输入和新增判断接进 runtime。

### 修改文件

* `skills/ll-dev-feat-to-tech/scripts/feat_to_tech_common.py`
* `skills/ll-dev-feat-to-tech/scripts/feat_to_tech_runtime.py`
* `skills/ll-dev-feat-to-tech/scripts/feat_to_tech_derivation.py`
* `skills/ll-dev-feat-to-tech/scripts/feat_to_tech_validation.py`
* `skills/ll-dev-feat-to-tech/agents/executor.md`
* `skills/ll-dev-feat-to-tech/agents/supervisor.md`

### 关键改造

* 在 `validate_input_package()` 中引入 `integration_context` 读取与结构校验
* 新增 `integration_sufficiency_check()`，校验：
  * legacy boundary 信息是否足够
  * canonical ownership 是否足够
  * compatibility / migration 是否足够
  * 是否还在依赖猜测性挂接
* 将 `assess_optional_artifacts()` 扩展为四项判断：
  * `arch_required`
  * `api_required`
  * `integration_context_sufficient`
  * `stateful_design_present`
* 将这四项判断及 rationale 写入 bundle
* supervisor 提示词补入：
  * 审核 `integration_context` 充分性
  * 审核状态机是否显式冻结
  * 未通过时强制 `revise`

### 完成标准

* 缺少关键信息的 `integration_context` 会在 freeze 前被拦下
* `integration_sufficiency_check` 与 `design_consistency_check` 同时进入 package

## Phase 3：`TECH` 十块结构与 package 序列化升级

### 目标

让输出 package 真正携带十块正式技术结构。

### 修改文件

* `skills/ll-dev-feat-to-tech/output/contract.yaml`
* `skills/ll-dev-feat-to-tech/output/schema.json`
* `skills/ll-dev-feat-to-tech/output/template.md`
* `skills/ll-dev-feat-to-tech/scripts/feat_to_tech_package_builder.py`
* `skills/ll-dev-feat-to-tech/scripts/feat_to_tech_documents.py`
* `skills/ll-dev-feat-to-tech/resources/examples/output.example.md`

### 关键改造

* 在 output schema 中新增并强制：
  * `integration_sufficiency_check`
  * `integration_context`
  * `need_assessment.integration_context_sufficient`
  * `need_assessment.stateful_design_present`
* 将 `tech_design` 结构扩展为十块，至少新增：
  * `state_machine`
  * `algorithm_constraints`
  * `io_matrix_and_side_effects`
  * `technical_glossary_and_canonical_ownership`
* 在模板中单独给出这些章节，禁止继续混写进 prose
* 在 bundle markdown / json 中把 `migration_constraints`、`canonical_owner_refs`、`integration_context_ref` 等引用固定下来

### 完成标准

* 生成的 `tech-spec.md` 与 `tech-design-bundle.json` 都能显式承载十块结构
* TECH 输出不再只靠抽象“Implementation Strategy”兜底

## Phase 4：handoff 扩展与下游 `tech_to_impl` 对齐

### 目标

阻断 `IMPL` 重新猜设计边界。

### 修改文件

* `skills/ll-dev-feat-to-tech/scripts/feat_to_tech_package_builder.py`
* `skills/ll-dev-feat-to-tech/scripts/feat_to_tech_cli_integration.py`
* `skills/ll-dev-tech-to-impl/SKILL.md`
* `skills/ll-dev-tech-to-impl/input/contract.yaml`
* `skills/ll-dev-tech-to-impl/input/schema.json`
* `skills/ll-dev-tech-to-impl/agents/executor.md`
* `skills/ll-dev-tech-to-impl/agents/supervisor.md`
* `skills/ll-dev-tech-to-impl/scripts/tech_to_impl_validation.py`

### 关键改造

* 扩展 `build_handoff()`，新增字段：
  * `integration_context_ref`
  * `canonical_owner_refs`
  * `state_machine_ref`
  * `nfr_constraints_ref`
  * `migration_constraints_ref`
  * `algorithm_constraint_refs`
* 在 `ll-dev-tech-to-impl` input contract/schema 中把这些字段纳入正式消费面
* 在 `tech_to_impl_validation.py` 中新增校验：
  * handoff 是否携带这些 refs
  * 若缺失是否属于允许的空值场景
  * `IMPL` 是否能据此直接进入实施组织而不是重做 TECH
* 更新下游 agent 提示词，明确不得改写状态机、算法、canonical owner 和 migration 语义

### 完成标准

* 下游输入验证能识别扩展 handoff
* `tech_to_impl` 不再依赖隐式上下文推断集成边界

## Phase 5：样例、文档与回归校验

### 目标

让新契约有可运行样例和回归验证。

### 修改文件

* `skills/ll-dev-feat-to-tech/resources/examples/input.example.md`
* `skills/ll-dev-feat-to-tech/resources/examples/output.example.md`
* `skills/ll-dev-tech-to-impl/resources/examples/input.example.md`
* 必要时补充 `ssot/tech/*` 或 `artifacts/` 下的代表性样例包

### 验证动作

* 运行 `skills/ll-dev-feat-to-tech/scripts/validate_input.sh`
* 运行 `skills/ll-dev-feat-to-tech/scripts/validate_output.sh`
* 运行 `skills/ll-dev-tech-to-impl/scripts/validate_input.sh`
* 选一个现有 `feat_freeze_package` 构造带 `integration_context` 的端到端样例，验证：
  * 上游生成新 bundle
  * handoff 字段齐全
  * 下游能通过输入验证

### 完成标准

* 至少一条 `feat-to-tech -> tech-to-impl` 样例链路跑通
* 新字段都出现在样例中，不是只存在于 schema

---

# 5. 推荐实施顺序

1. 先做 Phase 1 和 Phase 2。
2. 再做 Phase 3，让上游 bundle 真正产出新结构。
3. 紧接着做 Phase 4，保证 handoff 和下游同日对齐。
4. 最后做 Phase 5，补样例和回归验证。

原因很直接：

* 如果先改模板不改输入和 gate，生成物会空有章节没有治理含义。
* 如果先改 handoff 不改下游 schema，会造成链路断裂。
* 如果不补样例和校验，后续很容易回退到旧字段集合。

---

# 6. 交付清单

本次落地完成后，至少应交付：

* 升级后的 `ll-dev-feat-to-tech` input/output contract 与 schema
* 升级后的 `ll-dev-feat-to-tech` runtime、assessment、gate、bundle builder
* 升级后的 `ll-dev-tech-to-impl` input contract、schema 与 validation
* 一套体现 `integration_context` 和十块 `TECH` 结构的上游输出样例
* 一套能被下游消费的扩展 handoff 样例

---

# 7. 验收标准

当且仅当以下条件同时满足，才视为 ADR-013 修订完成落地：

1. 不带 `integration_context` 的输入无法再通过正式输入校验。
2. `tech-design-bundle.json` 中能稳定看到四项 need assessment。
3. `tech-design-bundle.json` 中能稳定看到 `integration_sufficiency_check`。
4. `tech-spec.md` 或等价结构化输出中能稳定看到十块 `TECH` 正式结构。
5. `handoff-to-tech-impl.json` 已携带新增的 integration / state / ownership / migration / algorithm refs。
6. `ll-dev-tech-to-impl` 能消费这些新增字段，且不会把缺失关键 ref 的 package 当成合格输入。

---

# 8. 风险与控制

## 风险 1：输入变重后，老样例全部失效

控制方式：

* 先补兼容样例
* 明确旧样例仅用于历史参考，不再作为正式 freeze-ready 输入

## 风险 2：十块 `TECH` 结构加入后，模板变重但 runtime 只填空话

控制方式：

* Phase 2 与 Phase 3 必须联动交付
* `stateful_design_present` 和 `integration_context_sufficient` 不能只靠模板存在来判真

## 风险 3：上游 handoff 扩展了，但下游仍按旧契约运行

控制方式：

* Phase 4 必须和上游 handoff 修改同批发布
* 下游 validation 对关键 ref 缺失直接 fail

---

# 9. 结论

这次改造不是重写 `feat-to-tech`，而是把它从“能产出一份技术设计包”升级为“能产出一份对现有系统集成语义、状态机语义、关键算法语义和 canonical ownership 负责的正式技术真相包”。

换句话说，改造完成后：

* 上游不再只回答“新能力应该怎么做”
* 而是正式回答“它怎样接进现在这套系统、哪些边界不能变、下游哪些东西不得重发明”
