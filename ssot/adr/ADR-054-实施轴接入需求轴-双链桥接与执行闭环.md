> **SSOT ID**: ADR-054
> **Title**: 通过 spec-to-TESTSET 桥接层将实施轴（test_exec_runtime）接入需求轴（ADR-047 spec 输出），打通 feat → spec → 执行 → settlement → gate 的完整流程
> **Status**: Draft
> **Version**: v1.1-draft (多角色评审修复版)
> **Effective Date**: TBD
> **Scope**: 测试治理 / 实施轴 / 需求轴桥接
> **Owner**: 架构 / QA 治理
> **Governance Kind**: NEW
> **Audience**: AI 实施代理、QA 技能编排层
> **Depends On**: ADR-047 (双链测试架构), ADR-048 (Droid Missions 接入), ADR-052 (测试体系轴化), ADR-053 (需求轴统一入口)

## 1. 背景

### 1.1 One-Sentence Summary

> **ADR-047 QA 链产出 `api-test-spec/*.md` 和 `e2e-journey-spec/*.md`，但废弃的 `ll-test-exec-cli`/`ll-test-exec-web-e2e` 强依赖 TESTSET YAML。通过 SPEC_ADAPTER_COMPAT 桥接格式 + environment-provision + test-orchestrator，打通需求轴 spec 输出到实施轴执行的完整链路。**

### 1.2 问题

#### 1.2.1 需求轴与实施轴之间存在断层

ADR-047 QA 链已完成需求轴资产建设：

```
feat-freeze-bundle
       ↓
ll-qa-feat-to-apiplan → api-test-plan.md
       ↓
ll-qa-api-manifest-init → api-coverage-manifest.yaml
       ↓
ll-qa-api-spec-gen → api-test-spec/*.md          ← 需求轴终点

e2e-journey-plan.md
       ↓
ll-qa-e2e-manifest-init → e2e-coverage-manifest.yaml
       ↓
ll-qa-e2e-spec-gen → e2e-journey-spec/*.md       ← 需求轴终点
```

**但这些 spec 文件没有下游消费者**：`ll-test-exec-cli` 和 `ll-test-exec-web-e2e` 已废弃（deprecated=true, 2026-04-14），且强依赖 TESTSET YAML（`test_exec_runtime.py:155` 强制 `ssot_type == TESTSET`）。

#### 1.2.2 `test_environment_ref` 无来源

ADR-052 §2.5 指出的 P0 模块 `environment-provision` 未实现。当前 `test_exec_runtime.py` 强制校验 `test_environment_ref` 存在且可解析，但无任何 Skill 负责生成该文件。

#### 1.2.3 两种架构路径的选择

| 路径 | 描述 | 决策 |
|------|------|------|
| **A：直接改 test_exec_runtime** | 让 test_exec_runtime 直接消费 spec 文件格式 | 改动大；影响现有 TESTSET 消费者；spec 字段与 case_expander 不匹配 |
| **B：spec-to-TESTSET 桥接层**（本文档） | spec 文件 → SPEC_ADAPTER_COMPAT 中间格式 → test_exec_runtime | 改动小；向后兼容；Phase 1 可快速验证 |

**决策：采用路径 B**。SPEC_ADAPTER_COMPAT 是临时桥接格式，待 ADR-048 Mission Compiler 实现后可替换。

### 1.3 ADR-053 关系

ADR-053 废弃了 `ll-qa-feat-to-testset`（TESTSET 策略层），但执行层（`test_units`）的内容仍需被 test_exec_runtime 消费。本文档补充执行层的桥接规范，与 ADR-053 形成互补：

- **ADR-053**：废弃 TESTSET 策略层，统一需求轴入口
- **本文档**：构建 spec → 执行层的桥接，打通实施轴

### 2.2 SPEC_ADAPTER_COMPAT 格式规范

**定位**：临时的 spec → TESTSET 中间格式。供 `test_exec_case_expander._build_case()` 消费。待 ADR-048 Mission Compiler 实现后，此格式被废弃。

#### 2.2.1 顶层结构

```yaml
ssot_type: SPEC_ADAPTER_COMPAT           # 固定值，test_exec_runtime 据此走兼容分支
test_set_id: spec-adapter-{feat_id}|{proto_id}
feat_ref: {feat_id}                      # API chain 时必填
prototype_ref: {proto_id}                # E2E chain 时必填
execution_modality: api|web_e2e|cli     # 决定走哪条执行路径
source_chain: api|spec_e2e                # 来源链标识
test_units: [...]                         # TESTSET 格式的 unit 列表
```

#### 2.2.2 API spec → TESTSET unit 字段映射

| API spec 字段 | TESTSET unit 字段 | 说明 |
|--------------|------------------|------|
| `case_id` | `unit_ref` | 直接映射 |
| `capability` + `scenario_type` | `title` | 拼接为 `"{capability}: {scenario_type}"` |
| `endpoint.method` + `endpoint.path` | `trigger_action` | 拼接为 `"{method} {path}"` |
| `assertions` 列表 | `pass_conditions` | 断言列表（去掉前缀 `!` 的项）|
| `assertions` 列表（带 `!` 前缀）| `fail_conditions` | 异常断言 |
| `preconditions` | `preconditions` | 直接映射 |
| `request` body dict | `test_data` | request JSON → dict |
| `source_feat_ref` | `acceptance_ref` | 追溯到 feat |
| `priority` | `priority` | 直接映射 |
| `evidence_required` | `required_evidence` | 直接映射 |

### 2.4 test_exec_runtime 兼容性修改

#### 2.4.1 `_validate_testset_execution_boundary()` 修改

在 `test_exec_runtime.py` 的 `_validate_testset_execution_boundary()` 函数中增加 `SPEC_ADAPTER_COMPAT` 分支：

```python
def _validate_testset_execution_boundary(test_set: dict[str, Any]) -> None:
    ssot_type = test_set.get("ssot_type", "")
    if ssot_type == "TESTSET":
        # 原有逻辑（向后兼容）
        ensure(test_set.get("ssot_type") == "TESTSET",
               "PRECONDITION_FAILED", "test_set_ref must resolve to a TESTSET")
        ensure(isinstance(test_set.get("test_units", []), list),
               "PRECONDITION_FAILED", "TESTSET must define test_units as strategy anchors")
        ensure(
            not bool(test_set.get("test_case_pack") or test_set.get("script_pack")),
            "PRECONDITION_FAILED",
            "TESTSET must not embed execution artifacts such as test_case_pack or script_pack",
        )
    elif ssot_type == "SPEC_ADAPTER_COMPAT":
        # 新增：spec adapter 格式兼容
        ensure(isinstance(test_set.get("test_units", []), list),
               "PRECONDITION_FAILED", "SPEC_ADAPTER_COMPAT must define test_units")
        ensure(
            bool(test_set.get("feat_ref") or test_set.get("prototype_ref")),
            "PRECONDITION_FAILED",
            "SPEC_ADAPTER_COMPAT must specify feat_ref or prototype_ref",
        )
    else:
        ensure(False, "PRECONDITION_FAILED",
               f"unsupported ssot_type: {ssot_type}")
```

#### 2.4.2 其他函数影响评估

| 函数 | 是否需修改 | 原因 |
|------|----------|------|
| `_validate_testset_execution_boundary()` | **是** | 增加 SPEC_ADAPTER_COMPAT 分支 |
| `_apply_testset_coverage_defaults()` | **否** | 读取 `environment` 字段，SPEC_ADAPTER_COMPAT 的 test_units 可被正常处理 |
| `_validate_environment()` | **否** | 只读 `environment` 参数 |

### 2.5 test_orchestrator 编排规范

#### 2.5.1 函数签名

```python
def run_spec_test(
    workspace_root: Path,
    *,
    feat_ref: str | None = None,     # API chain
    proto_ref: str | None = None,    # E2E chain
    base_url: str = "http://localhost:8000",
    modality: str = "api",            # "api" | "web_e2e" | "cli"
    coverage_mode: str = "smoke",
) -> dict[str, Any]:
    """
    端到端编排：
    1. environment-provision → ENV 文件
    2. spec_adapter → SPEC_ADAPTER_COMPAT 文件
    3. test_exec_runtime → 执行
    4. manifest_update → 更新 lifecycle_status + evidence_refs
    """
```

#### 2.5.2 编排流程

```
run_spec_test()
  │
  ├─ Step 1: provision_environment()
  │    ├─ 读取 feat/proto 的 environment_assumptions（如有）
  │    ├─ 合并 base_url + modality 参数
  │    ├─ 生成 ssot/environments/ENV-{id}.yaml
  │    └─ 返回 EnvConfig dataclass
  │
  ├─ Step 2: spec_to_testset()
  │    ├─ 解析 api-test-spec/*.md 或 e2e-journey-spec/*.md
  │    ├─ 按 2.2 映射规则生成 test_units
  │    ├─ 生成 ssot/tests/.spec-adapter/{id}.yaml
  │    └─ 返回 SPEC_ADAPTER_COMPAT dict（含 _source_ref）
  │
  ├─ Step 3: execute_test_exec_skill()
  │    └─ 调用 test_exec_runtime（action = test-exec-{modality}）
  │
  └─ Step 4: update_manifest()
       ├─ 读取 api-coverage-manifest.yaml 或 e2e-coverage-manifest.yaml
       ├─ 对每个 executed coverage_id，更新 lifecycle_status=executed
       ├─ 添加 evidence_refs = [artifact/candidate/*.json]
       └─ 落盘
```

## 4. 向后兼容性分析

### 4.1 TESTSET 路径完全不受影响

- `test_exec_runtime.py` 的 `ssot_type == "TESTSET"` 分支逻辑完全不变
- 现有使用 TESTSET 的流程继续工作
- SPEC_ADAPTER_COMPAT 是**额外分支**，不影响现有路径

### 4.2 manifest 更新机制

**ADR-047 要求**：spec 执行后，`api-coverage-manifest.yaml` 的对应 item 需更新：
- `lifecycle_status`：`designed` → `executed`
- `evidence_refs`：追加 `artifacts/active/qa/candidates/*.json`

**当前状态**：`test_exec_reporting.py` 的 `finalize_execution_outputs()` 有逻辑写入 evidence，但 **不直接更新 manifest 文件**。依赖手动调用 `ll-qa-settlement` 后才读取更新后的 manifest。

**本 ADR 决策**：`test_orchestrator` 在执行后显式调用 `update_manifest()`，确保 `ll-qa-settlement` 能读到最新状态。

## 5. 多角色评审修复（v1.1-draft）

> 以下为多角色评审（QA 架构师/AI 执行代理/测试工程实践者/端到端用户）发现的 CRITICAL 和 MAJOR 问题，已在 ADR 正文中修复。

### 5.1 CRITICAL 问题修复

#### R-1：Step 3 → Step 4 数据传递契约（CRITICAL）

**问题**：Step 3 `execute_test_exec_skill()` 的输出路径未传递给 Step 4 `update_manifest()`，导致 manifest 无法关联 coverage_id 与 evidence_refs。

**修复**：在 `StepResult` dataclass 中显式定义数据传递契约：

```python
@dataclass
class StepResult:
    """test_orchestrator 内部步骤之间的数据传递单元"""
    run_id: str
    execution_refs: dict[str, str]  # execution_outputs 中的 *_ref 字段
    candidate_path: str            # artifacts/active/qa/candidates/{slug}.json
    case_results: list[dict]      # 每个 case 的 coverage_id + case_id + status

    # manifest 更新需要的字段
    manifest_items: list[dict]     # [{coverage_id, status, evidence_ref}]
    execution_output_dir: str     # playwright output dir

def run_spec_test(...) -> StepResult:
    # Step 3 返回 StepResult（含 execution_refs 和 manifest_items）
    step3_result = execute_test_exec_skill(...)
    execution_refs = step3_result["refs"]

    # 构造 manifest_items：test_units[*] 包含源 coverage_id
    manifest_items = []
    for unit in test_units:
        coverage_id = unit.get("_source_coverage_id")  # spec_adapter 填充
        manifest_items.append({
            "coverage_id": coverage_id,
            "status": "executed",
            "evidence_ref": execution_refs.get("candidate_artifact_ref"),
        })

    # Step 4 接收 StepResult，显式消费 manifest_items
    update_manifest(workspace_root, manifest_items)
```

**spec_adapter.py 必须携带源 coverage_id**：

```python

unit["_source_coverage_id"] = spec["coverage_id"]   # 追溯到 manifest

# e2e spec → unit

```

#### R-2：E2E spec `user_steps[*].target` 格式规范化（CRITICAL）

**问题**：E2E spec 中 `user_steps[*].target` 格式未定义，`spec_adapter.py` 无法可靠构建 selectors。

**修复**：在 ADR-047 的 E2E spec 格式规范中增加 target_format 字段：

```yaml
user_steps:
  - action: "fill_minimum_profile"
    type: "form"
    target: ".profile-form"
    target_format: "css_selector"    # css_selector | xpath | semantic | text
    timeout: null
  - action: "submit"
    type: "interaction"
    target: "button[type=submit]"
    target_format: "css_selector"
    timeout: 30000
```

`spec_adapter.py` 根据 `target_format` 决定如何填入 `selectors`：

```python
def _resolve_selector(action: str, step: dict) -> str | None:
    fmt = step.get("target_format", "css_selector")
    if fmt == "css_selector":
        return step.get("target")  # 直接使用
    elif fmt == "semantic":
        return f"[data-action={action}]"  # 语义化 fallback
    elif fmt == "text":
        return f"text={step.get('target')}"  # playwright text locator
    return None  # 无法解析，留给 playwright probe
```

#### R-5：双链并行竞态控制（MAJOR）

**问题**：`--chain both` 时两个链同时写 manifest 存在竞态。

**修复**：manifest 文件锁机制：

```python
import fcntl

def update_manifest(workspace_root, manifest_items):
    manifest_path = workspace_root / f"ssot/tests/api/{feat_id}/api-coverage-manifest.yaml"
    with open(manifest_path, "r+") as f:
        fcntl.flock(f.fileno(), fcntl.LOCK_EX)
        try:
            manifest = yaml.safe_load(f)
            # 合并更新
            for item in manifest_items:
                for existing in manifest["items"]:
                    if existing["coverage_id"] == item["coverage_id"]:
                        existing["lifecycle_status"] = item["status"]
                        existing["evidence_refs"].extend(item["evidence_refs"])
            f.seek(0)
            yaml.safe_dump(manifest, f)
        finally:
            fcntl.flock(f.fileno(), fcntl.LOCK_UN)
```

**或者**：使用时间戳 + 版本号乐观锁（Windows 兼容）：

```python
def update_manifest_optimistic(workspace_root, manifest_items, expected_version: str):
    manifest_path = workspace_root / "..."
    manifest = yaml.safe_load(manifest_path.read_text())
    assert manifest["_version"] == expected_version, "CONCURRENT_MODIFICATION"
    # 更新...
    manifest["_version"] = str(uuid4())  # 乐观锁
    manifest_path.write_text(yaml.safe_dump(manifest))
```

def run_spec_test(workspace_root, *, resume: bool = False, resume_from: str | None = None):
