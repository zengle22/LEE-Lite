# ADR-052：测试体系轴化 — 需求轴与实施轴

> **SSOT ID**: ADR-052
> **Title**: 测试体系按需求轴/实施轴拆分，2 个用户入口 Skill 编排内部模块，SSOT 管"测什么"，Artifact 管"怎么跑、结果如何、是否可信"
> **Status**: Draft
> **Version**: v1.5-draft (第四轮修订：黄金路径语义边界修正/状态机失败处理说明/Skill命名一致性)
> **Effective Date**: TBD
> **Scope**: 测试治理 / 技能分层 / E2E 验收闭环
> **Owner**: 架构 / QA 治理
> **Governance Kind**: REFINEMENT
> **Audience**: AI 实施代理、QA 技能、Execution Loop Runner
> **Depends On**: ADR-047 (双链测试重建), ADR-050 (SSOT 语义治理总纲), ADR-051 (Task Pack 循环)

---

## 1. 背景

### 1.1 One-Sentence Summary

> **将测试体系拆分为需求轴（SSOT 管理，定义"测什么"）和实施轴（Artifact 管理，定义"在哪测、怎么跑、结果是否可信"），通过 2 个用户入口 Skill（qa.test-plan / qa.test-run）编排内部模块，补齐环境供给、Run Manifest、三层断言、状态机执行、独立验证、事故包等关键能力，形成从需求到验收的完整闭环。**

### 1.2 问题

当前测试体系存在两类问题：

#### 治理层问题

| 痛点 | 根因 |
|------|------|
| `test_environment_ref` 没有来源，代码强校验但无技能生成 | 环境文件不在任何技能职责范围内 |
| TESTSET（策略声明）和执行产物（环境、报告、Bug）混在同一治理域 | 没有区分"需求"和"实施"两类资产 |
| 用户需要手动创建 environment.yaml，但不知道该写什么格式 | 缺少环境定义规范 |

#### 执行层问题（E2E 验收失效）

用户旅程清晰、Playwright 环境可跑、AI 执行回报大量 case 成功，但手工验收主流程仍不通。症状指向 5 类叠加问题：

| 类别 | 症状 | 缺失能力 |
|------|------|----------|
| **环境不一致** | "它跑过了，为什么我这里不行？" | 无统一 Run Manifest 绑定版本/环境/数据 |
| **断言过弱** | 页面有 toast 就算成功，但业务实体未落库 | 无业务状态断言层 |
| **AI 执行不可审计** | AI 可能跳步、API 直调、误判 | 无状态机执行约束和违规检测 |
| **数据/前置不可控** | 同账号第二次跑失败、脏状态残留 | 无标准化测试数据供给和清理 |
| **失败排障困难** | 排查靠人工回放猜 | 无标准化事故包自动生成 |

**根因排序：**
1. 断言口径不对（测的是"页面操作完成"，不是"业务闭环完成"）
2. 环境与数据不一致（AI 跑的环境/账号/数据状态和手工验收不一致）
3. 异步流程未被正确等待（当时看起来成功，稍后失败或状态未真正落定）
4. AI 执行过程绕路（API 直调、跳步、误解释 UI）
5. 真实产品 bug

### 1.3 核心洞察

测试体系中有两类本质不同的资产，以及一个尚未被工程化强绑定的"执行可信度"问题：

| 维度 | 需求轴资产 | 实施轴资产 |
|------|-----------|-----------|
| **本质** | 声明（declarative） | 证据（evidential） |
| **问题** | "测什么？" | "在哪测？怎么跑？结果是否可信？" |
| **生命周期** | 随需求变更而更新 | 随执行产生而追加 |
| **治理方式** | SSOT 受治理对象 | Artifact 流水线产物 |
| **修改方式** | 通过技能重新编译 | 只追加，不修改 |
| **当前状态** | 基本完整 | 有骨架，缺关键能力 |

> **注意：** 本文中"轴"指技能职责的分野（需求 vs 实施），"链"指 ADR-047 定义的 API/E2E 并行测试通道。两者维度不同：每条链都有自己的需求轴资产和实施轴资产。

## 2. 决策

### 2.1 需求轴与实施轴拆分

```
需求轴（SSOT 管理，声明性）          实施轴（Artifact 管理，证据性）
┌──────────────────────────┐          ┌────────────────────────────────────┐
│  TESTSET                 │          │  [已有]                             │
│  ├─ api-plan             │          │  ├─ test-exec-web-e2e / cli        │
│  ├─ api-manifest         │          │  ├─ playwright 项目生成与执行       │
│  ├─ api-spec             │          │  ├─ case_pack 运行时展开            │
│  ├─ e2e-plan             │          │  ├─ UI 意图推导 + 选择器绑定        │
│  ├─ e2e-manifest         │          │  ├─ 证据收集（screenshot/trace）   │
│  └─ e2e-spec             │          │  ├─ 合规性检查 + 用例判定           │
│                          │          │  ├─ Bug bundle 生成                 │
│  存储：ssot/tests/       │          │  ├─ 覆盖率合并 + 报告               │
│  治理：SSOT 版本控制      │          │  ├─ Markdown 测试报告               │
│  更新：模块重新编译       │          │  ├─ settlement report               │
│  可覆盖：是               │          │  ├─ gate evaluation                 │
│                          │          │  └─ candidate artifact 落盘         │
│                          │          │                                     │
│                          │          │  [待补 — 实施轴能力缺口]              │
│                          │          │  ├─ environment-provision        │
│                          │          │  ├─ run-manifest-gen             │
│                          │          │  ├─ scenario-spec-compile        │
│                          │          │  ├─ state-machine-executor       │
│                          │          │  ├─ bypass-detector              │
│                          │          │  ├─ independent-verifier         │
│                          │          │  ├─ accident-package             │
│                          │          │  ├─ failure-classifier           │
│                          │          │  ├─ test-data-provision          │
│                          │          │  └─ l0-smoke-check               │
│                          │          │                                     │
│                          │          │  存储：artifacts/active/qa/         │
│                          │          │        ssot/tests/.artifacts/       │
│                          │          │  治理：Artifact 流水线               │
│                          │          │  更新：只追加                         │
└──────────────────────────┘          └────────────────────────────────────┘
```

### 2.2 用户入口 Skill（仅 2 个）

**设计原则：Skill 是用户意图的入口，不是内部能力模块的映射。** 用户只应关心"我要做什么"，不应关心内部拆了多少个模块。

| Skill ID | 用户意图 | 内部编排模块 | 输出 |
|----------|----------|-------------|------|
| `qa.test-plan` | "帮我把这个需求变成可执行的测试计划" | `feat-to-testset` → `api-plan-compile` → `api-manifest-compile` → `api-spec-compile` → `e2e-plan-compile` → `e2e-manifest-compile` → `e2e-spec-compile` | TESTSET + Plan/Manifest/Spec 全套 |
| `qa.test-run` | "在指定环境上跑一遍测试，给我结果和报告" | `environment-provision` → `test-data-provision` → `l0-smoke-check` → `run-manifest-gen` → `scenario-spec-compile` → `state-machine-executor` → `bypass-detector` → 已有执行层 → `accident-package` → `failure-classifier` → `independent-verifier` → 已有结算层 → `gate-evaluate` → `render-report` | 证据包 + Run Manifest + 测试报告 + Gate 结论 |

**交互增强：** `qa.test-plan` 支持 `--preview`（只生成不落盘，用户确认范围后再 `--apply`）和 `--target G1`（仅针对指定黄金路径生成计划）。`qa.test-run` 支持 `--last-failed`、`--report-only`、`--target G1`（仅跑指定黄金路径）。

**内部模块不注册为 Claude Skill**，仅作为 Python 模块被上述 2 个 Skill 编排调用。

**设计决策说明：** 将原 `qa.test-report` 合并入 `qa.test-run`。原因：用户跑完测试后自然想立刻看结果，拆成独立入口增加认知负担。`qa.test-run` 执行完毕后自动触发报告生成，同时支持 `--last-failed` 参数实现"只重跑失败用例"的高频场景。独立"只看已有报告不重跑"的需求通过 `qa.test-run --report-only` 选项满足，而非独立 Skill。

### 2.5 实施轴内部模块 — 待补能力

以下是实施轴从"脚本能跑"升级到"结果可信"必须补齐的内部模块（不注册为 Skill，由 `qa.test-run` 编排调用）：

#### 模块 1：`environment-provision` — 环境供给

**问题：** `test_environment_ref` 当前无来源。

```
输入：
  - TESTSET 的 environment_assumptions 和 required_environment_inputs
  - 用户提供的目标地址（base_url 或 command_entry）
  - 可选：browser, timeout 等偏好

输出：
  - ssot/environments/ENV-{feat-id}.yaml
  - 格式见 docs/guides/adr047-environment-spec.md

规则：
  - 不凭空猜测环境配置
  - 从 TESTSET 约束推导，用户确认目标地址
  - 生成后作为 test_environment_ref 的来源
```

#### 模块 2：`run-manifest-gen` — 运行清单生成

**问题：** 每次执行的环境/版本/数据未被记录，导致"它跑过的世界"和"你验收的世界"不同。

```
输入：
  - ENV 文件
  - 运行时信息（git sha, frontend build, backend version 等）
  - 测试账号、测试数据版本、feature flags

输出：
  - ssot/tests/.artifacts/e2e-runs/{run_id}/run-manifest.yaml

结构：
  run_id: E2E-RUN-20260418-001
  app_commit: abc123
  frontend_build: fe-20260418-02
  backend_build: be-20260418-01
  base_url: https://staging.xxx.com
  browser: chromium-135
  playwright: 1.52.0
  timezone: Asia/Shanghai
  locale: zh-CN
  feature_flags:
    use_new_plan_flow: true
  test_data_snapshot: seed-v17
  accounts:
    runner: e2e_runner_01
    verifier: e2e_verify_01
  generated_at: "2026-04-21T10:00:00Z"

规则：
  - 每次执行生成唯一 run_id
  - 所有测试结果绑定此 manifest
  - manifest 不可修改（只追加）
```

#### 模块 3：`scenario-spec-compile` — 场景规范编译

**问题：** 用户旅程是业务视角，不是执行视角。直接从用户旅程生成 Playwright 脚本导致 AI 漂移。

**方案：** 在用户旅程和可执行脚本之间插入 Scenario Spec 层：

```
用户旅程 → Scenario Spec → 可执行状态机脚本 → 验收规则 → 证据模板
```

```yaml
# Scenario Spec 结构
scenario_id: E2E-PLAN-001
title: 首次生成训练计划
preconditions:
  - account exists and has no active plan
  - onboarding not completed
inputs:
  age: 44
  weekly_days: 4
  target: 半马130
steps:
  - action: open_onboarding_page
    type: navigation
    target: "/onboarding"
  - action: fill_minimum_profile
    type: form
    fields: [age, weekly_days, target]
  - action: submit
    type: interaction
    target: "button:submit"
  - action: wait_for_plan_card
    type: assertion
    target: ".plan-card"
    timeout: 30000
expected_ui:
  - plan card visible
  - title contains 半马
  - weekly frequency equals 4
expected_state:
  - plan entity created
  - owner_id matches account
  - status = active
  - generated_at within 2 minutes
evidence:
  - screenshot_before_submit
  - screenshot_after_submit
  - trace
  - network_summary
  - entity_snapshot
```

**三层断言模型：**

| 断言层 | 证明什么 | 示例 | Phase 1-2 执行方式 | Phase 3+ 执行方式 |
|--------|----------|------|-------------------|------------------|
| **A. 交互断言** | UI 上真的做了这些动作 | 输入了什么、点了什么、页面跳到哪里 | Playwright `expect()` 在 UI 脚本中校验 | 同左 |
| **B. 页面结果断言** | 用户看见的结果是对的 | 计划卡片出现、标题/日期/强度正确、不是空壳骨架屏 | 精确 `expectedText` + 骨架屏检测（`assert_hidden: .skeleton-placeholder`） | 同左 |
| **C. 业务状态断言** | 系统内部状态也对了 | 数据库/API 返回的计划已创建、字段与 UI 一致、审计日志有对应动作 | HAR 捕获：拦截前端已知 API 响应，校验关键字段 | 独立 API 查询：verifier 用独立账号调后端 API 验证 |

**分阶段执行约束：**

| 阶段 | C 层要求 | 缺失处理 | 验证失败处理 |
|------|----------|----------|-------------|
| Phase 1 | C 层断言**不支持**（强制标记 `C_MISSING`） | 不计 failed，但 gate 评估视为条件不满足 | C 层存在但失败 → case 标记 failed |
| Phase 2 | 黄金路径必须通过 HAR 捕获满足 C 层 | 非黄金路径可标记 `C_MISSING` | 同左 |
| Phase 3+ | 黄金路径必须使用独立 API 查询路径 | 不满足 → gate 不得返回 pass | 同左 |

**C 层验证路径定义：**

| 路径 | 适用场景 | 实现成本 | 独立性 |
|------|---------|---------|--------|
| `har_capture` | 前端已知会调用 API，Phase 2 首选 | 低（HAR 解析 + `capture_api` step action） | 低（同会话共享认证） |
| `api` | 有 API 规范，runner 和 verifier 共享认证 | 中 | 中 |
| `independent_query` | 需独立账号/会话验证，Phase 3+ 首选 | 高（需 `api_client.py` + 独立认证） | 高 |
| `database` | 有只读 DB 访问（可选） | 高 | 最高 |

**关键约束：** 任何 Scenario Spec 必须定义 A、B 两层断言。C 层断言按阶段要求执行（见上表）。只证明 A+B 而 C 层缺失的 case，gate 评估时标记为 `conditional_pass` 而非 `pass`。

#### 模块 4：`state-machine-executor` — 状态机执行器

**问题：** AI 自由发挥执行测试，可能跳步、误判、脏状态续跑。

**方案：** 把每条用例拆为有限状态节点，每个节点有明确的输入、允许动作、必须证据、成功判定、失败退出码：

```
状态机节点定义：
┌─────────────┬──────────────┬──────────────┬──────────────┬──────────────┐
│ 状态         │ 前置条件      │ 允许动作      │ 必须产出证据  │ 成功判定      │
├─────────────┼──────────────┼──────────────┼──────────────┼──────────────┤
│ SETUP       │ 环境就绪      │ 创建测试账号  │ 账号ID        │ 账号可登录    │
│ LOGIN       │ 账号存在      │ 输入凭证/登录 │ 登录截图      │ 进入首页      │
│ NAVIGATE    │ 已登录        │ 导航到目标页  │ URL/截图     │ 目标页可见    │
│ INPUT_FORM  │ 目标页打开    │ 填写表单      │ 填写前截图    │ 表单完成      │
│ SUBMIT      │ 表单完成      │ 点击提交      │ 提交前后截图   │ 触发目标请求  │
│             │              │              │ + 网络请求摘要 │ + 返回码正确  │
│             │              │              │ + console摘要 │ + 页面进入目标│
│ WAIT_EFFECT │ 提交完成      │ 轮询/等待    │ 轮询日志      │ 目标状态出现  │
│ VERIFY_UI   │ 等待完成      │ 页面断言      │ 验证截图      │ 所有UI断言通过│
│ VERIFY_STATE│ UI验证通过    │ 独立查询      │ 查询结果      │ 业务状态一致  │
│ COLLECT     │ 验证完成      │ 采集证据      │ 全量证据包    │ 证据完整      │
│ DONE        │ 证据完整      │ —            │ —            │ —            │
└─────────────┴──────────────┴──────────────┴──────────────┴──────────────┘

失败处理：任何节点失败 → 记录失败状态 → 跳出执行 → 标记失败节点 → 生成事故包
```

#### 模块 5：`bypass-detector` — 违规检测

**问题：** AI/Playwright 可能绕过 UI 直接调 API 创建实体，产生"假 UI 测试"。

```
输入：
  - Scenario Spec 中定义的执行模式
  - Playwright 执行期间的网络请求记录（HAR）
  - 允许/禁止动作策略定义

输出：
  - bypass-violations.json

检测规则：
  允许（UI E2E 模式）：
    - page.goto, click, fill, press, select
    - locator assertions
    - screenshot, trace

  禁止（UI E2E 模式）：
    - 直接调用业务 API 创建核心实体
    - 通过脚本注入伪造状态
    - 直接改 localStorage 伪造登录态
    - 用后门接口跳过关键交互

判定：
  - 如果"创建计划"的 UI 流程中没有出现预期前端网络调用链
  - 或出现了异常直调 → 直接 fail + 记录违规
```

#### 模块 6：`independent-verifier` — 独立验证

**问题：** AI 自己跑脚本、自己解释截图、自己总结"成功"——这是同源自证，不是验证。

```
输入：
  - runner 产出的完整证据包
  - 独立的验证查询（独立会话、独立账号、独立查询路径）

职责：
  - 不直接相信 runner 的结论
  - 只相信产物
  - 独立复核业务状态
  - 检查证据完整性
  - 扫描异常日志
  - 检测是否命中禁止模式

输出：
  - verifier-report.json / verifier-report.md
  - verdict: pass | conditional_pass | fail
  - 复核详情和置信度

关键约束：
  - 验证层与执行层必须分离
  - verifier 不执行 UI 操作，只检查已有产物 + 独立查询业务状态
```

#### 模块 7：`accident-package` — 事故包生成

**问题：** 失败后没有统一取证，排查只能靠人工回放猜。

```
输入：
  - 失败的 case 运行结果
  - Playwright 产出（trace, video, screenshots, network, console）

输出：
  - ssot/tests/.artifacts/e2e-runs/{run_id}/accident-{case_id}/
    ├── manifest.yaml           # 运行清单
    ├── case-summary.json       # 用例摘要
    ├── step-logs/              # 步骤日志
    ├── screenshots/            # 关键截图
    ├── traces/                 # Playwright trace
    ├── videos/                 # 录制视频
    ├── network/                # HAR / 请求摘要
    ├── console/                # 浏览器 console
    ├── storage-state/          # cookie / localStorage
    ├── assertions/             # 断言结果
    └── failure-classification.yaml  # 故障分类

标准事故包内容：
  - Playwright trace
  - 关键步骤截图（提交前/提交后/失败时）
  - 视频
  - 浏览器 console log
  - network HAR 或请求摘要
  - 最终 DOM snapshot
  - storage / cookie / localStorage 状态
  - 后端关键接口返回
  - 测试数据快照
  - 失败时最后一步的状态说明
```

#### 模块 8：`failure-classifier` — 故障分类

**问题：** 每次失败归因都从头猜，没有标准化分类。

```
输入：
  - 事故包内容
  - runner 和 verifier 报告

输出：
  - failure-classification.yaml

8 类故障模型：
| 类别      | 含义            | 常见表现              |
|-----------|-----------------|-----------------------|
| ENV       | 环境不一致/服务异常 | 手工验收不复现、偶发 404/500 |
| DATA      | 测试数据脏/前置不满足 | 同账号第二次跑失败、脏状态残留 |
| SCRIPT    | 脚本选择器/等待条件问题 | AI 说成功，实际点错/没等到 |
| ORACLE    | 断言太弱          | 页面有 toast 就算成功 |
| BYPASS    | 绕过 UI / 非法捷径  | 调 API、伪造状态 |
| PRODUCT   | 产品真实 bug      | 人手工也跑不通 |
| **FLAKY** | 非确定性/时序竞争  | 偶发失败、异步未稳定、随机因素 |
| **TIMEOUT** | 业务处理超时     | AI 生成计划超过等待上限、服务端响应慢 |

格式：
  failure_classification:
    primary: ORACLE
    secondary: SCRIPT
    confidence: medium
    notes: "脚本以 toast 为通过条件，但计划实体未落库"

故障分类后处理规则：
  - PRODUCT 类故障 → 自动触发回归测试用例创建流程，新 case 加入 TESTSET
  - SCRIPT/ORACLE 类故障 → 标记对应 Scenario Spec 需更新，下次 qa.test-run 强制执行
  - FLAKY 类故障 → 自动重跑 N 次确认，确认非偶发后才计入统计
  - 故障趋势度量：按时间维度统计 8 类故障分布变化，评估测试体系改进效果
```

#### 模块 9：`test-data-provision` — 测试数据供给

**问题：** 测试数据和前置状态不可控，导致偶现/漂移。

```
输入：
  - Scenario Spec 的 preconditions 和 inputs
  - 环境配置

输出：
  - 测试数据快照（seed data）
  - 数据版本标识
  - 数据清理策略

职责：
  - 根据 preconditions 准备测试数据
  - 记录数据版本号（test_data_snapshot）
  - 执行前验证数据就绪
  - 执行后清理/回滚
```

#### 模块 10：`l0-smoke-check` — L0 环境冒烟

**问题：** 服务可能都没通，AI 就开始跑 E2E，浪费时间且结果不可信。

```
输入：
  - ENV 文件
  - 冒烟检查清单

检查项：
  - 服务存活（base_url 可访问）
  - 关键接口通
  - 前端可访问
  - 登录可用
  - 基础数据存在
  - 队列/任务系统正常

输出：
  - smoke-check-result.yaml: pass | fail
  - L0 失败 → 不进入后续 E2E

分层执行模型：
  L0: 环境冒烟 → 服务是否值得测试
  L1: 页面交互冒烟 → UI 壳层是否正常
  L2: 主流程业务 E2E → 业务闭环是否成立
  L3: 验收复核 → 结果是否可信
```

### 2.8 存储策略与清理

> **背景：** 架构评审发现 `.artifacts/` 放在 `ssot/tests/` 内造成语义边界模糊，且产物膨胀可能导致 IDE 扫描变慢和潜在的 git 风险。

| 轻量实施产物 | `ssot/tests/.artifacts/settlement/`, `.artifacts/gate/` | 是 | 轻量 YAML，可安全重建 |
| 重执行产物 | `ssot/tests/.artifacts/e2e-runs/` | 是 | trace/video/HAR/screenshot |
| 重执行产物（Phase 2+） | `artifacts/active/qa/e2e-runs/` | 是 | 从 ssot 迁移至此 |

#### 2.8.2 .gitignore 规范

```gitignore
# ssot 内部实施轴产物

# 重文件类型级保护（防意外提交）

```

#### 2.8.3 清理策略

| 事故包（trace/video/screenshots） | 保留 30 天，之后仅保留 `case-summary.json` + `failure-classification.yaml` | 重文件占用大，摘要信息足够 |

|------|------|
| **立即** | 加强 `.gitignore`（见上），在 `ssot/tests/.artifacts/` 下创建 `README.md` 标注运行时产物 |

### 2.9 编排与执行引擎设计

> **背景：** 架构评审发现，如果在 Skill prompt 中仅用自然语言描述 8+ 模块的执行顺序，LLM 的随机性会导致执行顺序、错误处理、数据传递不可靠。本节定义各阶段的编排方案。

#### 2.9.1 编排原则

| 原则 | 说明 |
|------|------|
| 模块不注册为 Skill | 内部模块由 Python 直接调用，不暴露给用户 |
| 结构化接口通信 | 模块间通过 dataclass/dict schema 传递数据，不通过自然语言 |
| 依赖显式声明 | Phase 3 后使用 YAML DAG 定义模块间依赖关系 |
| 状态机在模块内 | `state-machine-executor` 的 9 个状态在模块内部实现为有限状态机，不推给编排层 |

#### 2.9.2 分阶段编排方案

| 阶段 | 编排方式 | 理由 | 文件位置 |
|------|---------|------|---------|
| Phase 1-2 | **函数链（Function Chain）** | 只有 2-4 个模块，线性依赖，直接按序调用 | `cli/lib/test_exec_orchestrator.py` |
| Phase 3+ | **轻量 DAG Runner** | bypass-detector 需并发监控 HAR，实施轴总模块数超过 10 个，函数链难以维护 | `cli/lib/dag_runner.py` + `test_run_dag.yaml` |
| 不采用 | 全功能编排器（重试/回滚/分布式） | 过度工程化，当前不需要分布式能力 | — |

#### 2.9.3 Phase 1-2 函数链模式

```python
# cli/lib/test_exec_orchestrator.py — Phase 1 示例
def run_test_run_pipeline(workspace_root: str, test_set_ref: str, user_prefs: dict) -> dict:
    """qa.test-run 的线性编排（Phase 1-2 使用）"""
    # Step 1: 环境供给
    env = environment_provision(workspace_root, test_set_ref, user_prefs)
    if not env:
        return {"status": "failed", "reason": "environment_provision_failed"}

    # Step 2: L0 冒烟检查
    smoke = l0_smoke_check(env)
    if smoke["status"] != "pass":
        return {"status": "stopped", "reason": "l0_smoke_failed", "smoke": smoke}

    # Step 3: Run Manifest 生成
    manifest = run_manifest_gen(workspace_root, env)

    # Step 4: Scenario Spec 编译
    spec = scenario_spec_compile(test_set_ref, manifest, user_prefs)

    # Step 5: 状态机执行 + bypass-detector（Phase 1 函数链降级：执行后分析）
    executor_result, evidence, har = run_executor_with_bypass_check(spec, env)
    if executor_result.status != "pass":
        # Step 5a: 事故包
        accident = accident_package(evidence, executor_result)
        # Step 5b: 故障分类
        classification = failure_classify(accident)

    # Step 6: 独立验证
    verifier_report = independent_verify(evidence, spec, env)

    # Step 7: 结算 + Gate
    settlement = evaluate_settlement(evidence, verifier_report)
    gate = evaluate_gate(settlement, verifier_report, evidence)

    # Step 8: 报告渲染
    return render_report(gate, settlement, verifier_report, evidence)
```

#### 2.9.4 Phase 3+ 轻量 DAG Runner

```yaml
# cli/lib/test_run_dag.yaml — qa.test-run 的 DAG 定义（Phase 3+）
pipeline: qa.test-run
nodes:
  - id: environment_provision
    module: environment_provision
    inputs: { test_set: "{{ test_set }}", user_prefs: "{{ user_prefs }}" }
    outputs: [env]

  - id: l0_smoke_check
    module: l0_smoke_check
    inputs: { env: "{{ environment_provision.env }}" }
    outputs: [smoke_result]
    gate: "smoke_result.status == 'pass'"

  - id: run_manifest_gen
    module: run_manifest_gen
    inputs: { env: "{{ environment_provision.env }}" }
    depends_on: [l0_smoke_check]
    outputs: [manifest]

  - id: scenario_spec_compile
    module: scenario_spec_compile
    inputs: { test_set: "{{ test_set }}", manifest: "{{ run_manifest_gen.manifest }}" }
    depends_on: [run_manifest_gen]
    outputs: [spec]

  - id: state_machine_executor
    module: state_machine_executor
    inputs: { spec: "{{ scenario_spec_compile.spec }}", env: "{{ environment_provision.env }}" }
    depends_on: [scenario_spec_compile]
    outputs: [executor_result, evidence_bundle]

  - id: bypass_detector
    module: bypass_detector
    inputs: { spec: "{{ scenario_spec_compile.spec }}", har_stream: "{{ state_machine_executor.har_stream }}" }
    depends_on: [scenario_spec_compile]
    trigger: parallel_with
    parallel_with: [state_machine_executor]
    outputs: [violations]

  - id: web_e2e
    module: test_exec_web_e2e
    inputs: { spec: "{{ scenario_spec_compile.spec }}", env: "{{ environment_provision.env }}" }
    depends_on: [scenario_spec_compile]
    outputs: [case_runs]
```

#### 2.9.5 bypass-detector 并发模式

`bypass-detector` 不是"执行后检测"，而是"执行中监控"：

```
[启动] state-machine-executor + bypass-detector(HAR capture) 并发运行
[等待] state-machine-executor 完成
[分析] bypass-detector 分析 HAR → violations
```

**DAG 语义：** bypass-detector 节点使用 `trigger: parallel_with`（而非默认 `depends_on` 的"前驱完成后启动"语义），表示与 `state_machine_executor` 同时启动。Phase 1-2 函数链阶段降级为"执行后分析 HAR"（`run_executor_with_bypass_check` 单一函数调用内部启动监控线程）。

#### 2.9.6 错误传播规则

定义全局错误传播策略，所有模块遵循统一的结构化接口。各 dataclass 定义在 `cli/lib/contracts.py`：

```python
# cli/lib/contracts.py — 模块间通信 dataclass（Phase 1 实现前必须完成）

@dataclass
class EnvConfig:
    """environment-provision 输出"""
    base_url: str
    browser: str            # chromium | firefox | webkit
    timeout: int
    headless: bool
    account_runner: str     # runner 测试账号
    account_verifier: str   # verifier 独立测试账号
    env_file_path: str      # 生成的 ENV YAML 路径

@dataclass
class SmokeResult:
    """l0-smoke-check 输出"""
    status: str             # pass | fail
    checks: list[dict]      # [{name, status, detail}]

@dataclass
class RunManifest:
    """run-manifest-gen 输出"""
    run_id: str
    app_commit: str
    frontend_build: str | None
    backend_version: str | None
    env_ref: str            # EnvConfig.base_url
    test_data_version: str | None
    generated_at: str

@dataclass
class ScenarioSpec:
    """scenario-spec-compile 输出"""
    scenario_id: str
    steps: list[dict]       # 每步包含 action + assertions(A/B/C)
    preconditions: list[str]
    skip_states: list[str]  # 状态机跳路配置
    assertion_layers: dict  # {a: bool, b: bool, c: bool}

@dataclass
class ExecutionResult:
    """state-machine-executor 输出"""
    status: str                  # pass | fail | conditional_pass | blocked
    failure_node: str | None     # 哪个状态节点失败
    failure_classification: str  # ENV | DATA | SCRIPT | ORACLE | BYPASS | PRODUCT | FLAKY | TIMEOUT
    evidence_refs: list[str]     # 关联的证据路径
    continue_allowed: bool       # 是否允许继续后续节点
    evidence_incomplete: bool    # COLLECT on_fail 时标记（Phase 1 新增）
    har_path: str | None         # HAR 文件路径（供 bypass-detector 消费）

@dataclass
class VerifierReport:
    """independent-verifier 输出"""
    verdict: str            # pass | conditional_pass | fail
    confidence: str         # high | medium | low
    c_layer_verdict: str    # verified | C_MISSING | failed
    detail: dict
```

# 传播规则：
# - L0 失败 → 整个 pipeline 停止，返回 l0_failed
# - state-machine 任一状态失败 → 进入 COLLECT 收集证据 → 生成事故包 → 返回 state_machine_failed_at:{node}
# - bypass 违规 → 自动 fail，但继续完成执行以收集完整证据
# - verifier 与 runner 结论冲突 → 以 verifier 为准
```

#### 2.9.7 模块接口契约

每个模块定义输入输出的最小 schema：

| 模块 | 输入 | 输出 |
|------|------|------|
| `environment-provision` | `test_set.environment_assumptions`, `target_address`, `preferences` | `env_file` (YAML), `EnvConfig` dataclass (`base_url`, `browser`, `timeout`, `account_runner`, `account_verifier`) |
| `l0-smoke-check` | `EnvConfig` | `SmokeResult` dataclass (`status: pass/fail`, `checks[]`) |
| `run-manifest-gen` | `EnvConfig`, runtime info | `RunManifest` dataclass + YAML |
| `scenario-spec-compile` | `test_set`, `RunManifest` | `ScenarioSpec` dataclass (含三层断言结构化定义，定义见 `cli/lib/contracts.py`) |
| `state-machine-executor` | `ScenarioSpec`, `EnvConfig` | `ExecutionResult`, `evidence_bundle`, `har` |
| `bypass-detector` | `ScenarioSpec`, `har` | `violations[]` |
| `independent-verifier` | `evidence_bundle`, `ScenarioSpec`, `EnvConfig` | `VerifierReport` dataclass (`verdict`, `confidence`) |

#### 2.9.8 状态机内部转换表

`state-machine-executor` 的 9 个状态在模块内部实现为有限状态机：

```python
STATE_TRANSITIONS = {
    "SETUP":        {"allowed_next": ["LOGIN"],         "on_fail": "HALT"},
    "LOGIN":        {"allowed_next": ["NAVIGATE"],      "on_fail": "HALT"},
    "NAVIGATE":     {"allowed_next": ["INPUT_FORM"],    "on_fail": "HALT"},
    "INPUT_FORM":   {"allowed_next": ["SUBMIT"],        "on_fail": "HALT"},
    "SUBMIT":       {"allowed_next": ["WAIT_EFFECT"],   "on_fail": "HALT"},
    "WAIT_EFFECT":  {"allowed_next": ["VERIFY_UI"],     "on_fail": "HALT"},
    "VERIFY_UI":    {"allowed_next": ["VERIFY_STATE"],  "on_fail": "COLLECT"},
    "VERIFY_STATE": {"allowed_next": ["COLLECT"],       "on_fail": "COLLECT"},
    "COLLECT":      {"allowed_next": ["DONE"],          "on_fail": "DONE"},
    "DONE":         {"allowed_next": [],                "on_fail": "DONE"},
}
```

**动态子集支持：** Scenario Spec 可指定需要哪些状态，状态机跳过不适用节点（如已登录用户跳过 LOGIN）。状态机启动时根据 Spec 计算有效转换路径：

```python
# Phase 1 简化版 3 状态映射（对应完整版子集）
PHASE1_STATE_TRANSITIONS = {
    # SETUP = SETUP + LOGIN + NAVIGATE + INPUT_FORM + SUBMIT
    # EXECUTE = WAIT_EFFECT
    # VERIFY = VERIFY_UI + VERIFY_STATE
    # COLLECT + DONE 保持不变
    "SETUP":    {"allowed_next": ["EXECUTE"], "on_fail": "COLLECT",
                 "covers": ["SETUP", "LOGIN", "NAVIGATE", "INPUT_FORM", "SUBMIT"]},
    "EXECUTE":  {"allowed_next": ["VERIFY"],  "on_fail": "COLLECT",
                 "covers": ["WAIT_EFFECT"]},
    "VERIFY":   {"allowed_next": ["COLLECT"], "on_fail": "COLLECT",
                 "covers": ["VERIFY_UI", "VERIFY_STATE"]},
    "COLLECT":  {"allowed_next": ["DONE"],    "on_fail": "DONE",
                 "evidence_incomplete_on_fail": True},
    "DONE":     {"allowed_next": [],          "on_fail": "DONE"},
}
# 
# 设计说明：Phase 1 简化版所有非 DONE 状态失败都进入 COLLECT。
# 原因：简化版目标是"最小可运行验证"，即使前置阶段失败也应收集已产生的证据，
# 以便调试框架本身的问题。完整版则区分"前置失败(HALT)"和"验证失败(COLLECT)"。
# Phase 3 切换到完整版后，SETUP/LOGIN 等前置失败会直接 HALT，不再收集证据。
```

**跳路机制：** Scenario Spec 的 `skip_states` 参数在启动时被解析为有效路径。例如 `skip_states: ["LOGIN"]` 时，完整版的 SETUP 直接允许跳转到 NAVIGATE。Phase 1 的简化版默认包含所有前置步骤，不支持跳路。

**Phase 3 扩展策略：** Phase 1 使用 `PHASE1_STATE_TRANSITIONS`，Phase 3 切换到 `STATE_TRANSITIONS`。两者通过统一的 `StateNode` 抽象（`execute()`, `verify()`, `collect_evidence()`）共享接口，切换方式为配置文件开关（`state_machine.mode: simplified | full`）。

### 3.1 端到端流程

```
用户入口：qa.test-plan（阶段1：需求轴）
  FEAT 文档 (ssot/features/)
    ↓
  feat-to-testset
    ├─→ TESTSET.yaml (ssot/testset/)
    │   定义: test_units, coverage_scope, environment_assumptions
    │

    ├─→ api-plan.yaml → api-manifest.yaml → api-spec.md
    │
  e2e-plan-compile + e2e-manifest-compile + e2e-spec-compile
    ├─→ e2e-plan.yaml → e2e-manifest.yaml → e2e-spec.md

用户入口：qa.test-run（阶段2-3：实施执行）
  [阶段2: 实施准备]
  environment-provision
    输入: TESTSET (environment_assumptions) + 用户提供目标地址
    ├─→ ENV-{feat-id}.yaml (ssot/environments/)

  test-data-provision
    输入: Scenario Spec (preconditions + inputs)
    ├─→ 测试数据快照 + 数据版本标识

  [阶段3: 实施执行 — Artifact 管理]
  run-manifest-gen
    ├─→ run-manifest.yaml (绑定版本/环境/账号/数据)
    │
  scenario-spec-compile
    输入: Manifest / Spec
    ├─→ Scenario Spec (三层断言: 交互 + 页面结果 + 业务状态)
    │
  state-machine-executor (替代当前自由执行模式)
    输入: Scenario Spec + ENV
    ├─→ 状态机逐节点执行: SETUP → LOGIN → NAVIGATE → ... → VERIFY_STATE → DONE
    ├─→ 每节点产出: 前置检查 → 允许动作 → 必须证据 → 成功判定 / 失败退出
    │
  bypass-detector (并行监控)
    ├─→ bypass-violations.json (检测 API 直调、伪造状态等)
    │
  [Playwright 执行层 — 已有]
    ├─→ case_pack 展开
    ├─→ UI 意图推导 + 选择器绑定
    ├─→ Playwright 项目生成与执行
    └─→ 证据收集 (screenshot/trace/result.json)

qa.test-run（阶段4-5：验证与结算 — 合并自原 qa.test-report）
  [阶段4: 验证与排障]
  accident-package (失败时自动触发)
    ├─→ 标准事故包: trace + 截图 + 视频 + network + console + storage
    │
  failure-classifier
    输入: 事故包 + runner/verifier 报告
    ├─→ failure-classification.yaml (8类: ENV/DATA/SCRIPT/ORACLE/BYPASS/PRODUCT/FLAKY/TIMEOUT)
    │
  independent-verifier (与 runner 分离)
    输入: runner 完整证据包
    ├─→ 独立复核业务状态（独立查询路径）

  [阶段5: 结算与 Gate — 已有模块]
    ├─→ 合规性检查 + 用例判定
    ├─→ Bug bundle 生成
    ├─→ 覆盖率合并 + 报告
    ├─→ Markdown 测试报告
    ├─→ settlement report
    └─→ gate evaluation (pass/conditional_pass/fail)
```

### 3.3 触发条件

| 触发 | 需要重新执行的模块链 |
|------|---------------------|
| FEAT 需求变更 | 变更影响分析 → 判定受影响模块 → 从受影响点重新编译/执行 |
| TESTSET 变更 | 从 TESTSET 之后的需求轴模块重新编译 + 实施轴重新执行 |

| 重跑/仅失败用例 | 仅失败 case + 关联 Scenario Spec 重新执行 |

> **变更影响分析：** 全量重跑前先判断哪些模块的输出实际发生了变更。引入 `change_hash` 字段对比上次执行，如 FEAT 变更仅涉及文案调整且实施轴产物仍有效，则跳过全量重跑。

## 4. 实施优先级

### Phase 1a：最小可运行验证（2 周内）

> **目标：** 一条黄金路径能从"有测试计划"到"跑完并观察到结果"。

| 优先级 | 模块 | 原因 |
|--------|------|------|
| P0 | `scenario-spec-compile`（最小结构：steps + A层断言） | 从 e2e-spec.md 到 Scenario Spec 的最小可消费格式 |
| P0 | `state-machine-executor`（简化版 3 状态） | 把 AI 从自由发挥改为受控执行 |
| P0 | 手动 ENV 提供（不自动化） | 环境格式未稳定，先验证"能跑"再"自动供给" |
| P0 | `run-manifest-gen`（最小：run_id + base_url + generated_at） | 最小世界绑定 |

> **Phase 1a 已知限制：** 测试数据无隔离（同账号第二次跑可能失败，需手动清理）；环境需手动提供；C 层断言不支持。

### Phase 1b：三层断言 + 临时 Gate（再 2 周）

> **目标：** 结果基本可信，三层断言 A+B 层跑通，`provisional_pass` Gate 可用。

| 优先级 | 模块 | 原因 |
|--------|------|------|
| P0 | `scenario-spec-compile` 自动化（从 e2e-spec.md 自动解析） | 减少手工 Spec 编写 |
| P0 | B 层断言跑通 | 页面结果验证是结果可信的底线 |
| P0 | 完整 manifest + `provisional_pass` gate | 透明披露 C 层缺口 |
| P0 | `environment-provision` | 环境格式基本稳定，可自动化生成 |
| P0 | `e2e-spec-compile`（需求轴，使 Scenario Spec 有来源可消费） | 需求轴到实施轴的衔接 |

> **首批验证对象选择（示例）：** 测试框架验证时需要选择具体的黄金路径。示例产品中，G1（训练反馈调整）的业务前置条件是 G2/G3，因此验证顺序受业务依赖约束。Phase 1a 选择 G2+G1 作为首批验证对象（假设新用户已完成 onboarding），Phase 1b 扩展到 G3 覆盖完整首次体验。**说明：这是验证对象选择策略，不是测试框架模块的依赖关系。**

### Phase 2：独立验证（P0）+ 排障（P1）+ 冒烟（P1）

> **评审决策：** `independent-verifier` 升为 P0（没有独立验证，"AI 说通过了但实际没过"的问题依然存在），`l0-smoke-check` 升为 P1，`accident-package` + `failure-classifier` 升为 P1。

| 优先级 | 模块 | 原因 |
|--------|------|------|
| P0 | `independent-verifier` | runner 和 verifier 分离，消除同源自证 — **没有就不信** |
| P1 | `l0-smoke-check` | 服务不通时不浪费 AI 执行 — 先确保"服务值得测试" |
| P1 | `accident-package` | 失败后不再从零开始排查 |
| P1 | `failure-classifier` | 快速看出主要矛盾是什么，Phase 2 强化断言后必然出现大量之前被掩盖的失败 |

### Phase 3：稳健性 + 编排升级

| 优先级 | 模块 | 原因 |
|--------|------|------|
| P2 | `bypass-detector` | 状态机已约束执行路径后，bypass 窗口已变窄 — 降级为"审计保险" |
| P2 | `test-data-provision` | 消除数据脏/前置不满足导致的偶现失败 |
| P2 | `dag-runner` | 编排引擎升级：从函数链到 YAML DAG |

### Phase 4：需求轴 Skill + 全面切换

| 优先级 | 模块 | 原因 |
|--------|------|------|
| — | `qa.test-plan` Skill | 编排需求轴全部模块 |

## 6. 黄金路径验证策略

> **语义边界说明：** 黄金路径（Golden Path）是**被测对象**，由产品侧定义。测试框架定义 GoldenPath 的 schema 结构和验证标准，但不定义具体路径内容。本节以示例性产品业务流程（G1-G6）说明验证策略的运作方式。

### 验证策略原则

不追求大而全，先聚焦 **3~5 条主流程黄金路径**，用它们验证测试框架的"结果可信"能力是否达到硬标准。

### 示例性黄金路径（产品侧定义）

> **说明：** 以下 G1-G6 是示例性产品业务流程，用于说明测试框架如何选择首批验证对象。具体内容由产品侧定义，排序以**产品核心价值主张**为第一优先级。

| 标识 | 业务流程（示例） | 产品侧排序理由 |
|------|-----------------|---------------|
| G1 | 用户完成一次训练反馈后计划被正确调整 | **产品核心价值主张** — 如果这个链路不硬，其他路径再硬也没用 |
| G2 | 成功生成首版训练计划 | G1 的业务前置条件 |
| G3 | 新用户最小信息采集成功 | G2 的业务前置条件，但业务逻辑最简单 |
| G4 | 用户中断后恢复 | **真实高频场景** |
| G5 | 异常输入下系统给出受控反馈 | 质量底线 |
| G6 | 历史记录可再次打开且数据一致 | 留存用户体验保障 |

### 黄金路径验证标准（测试框架定义）

> **说明：** 以下标准是测试框架定义的验证能力要求，适用于所有黄金路径。

每条黄金路径必须满足：
- [ ] 固定环境（Run Manifest 唯一绑定）
- [ ] 固定数据（测试数据版本化）
- [ ] 固定账号（runner + verifier 独立账号）
- [ ] 固定步骤（状态机逐节点执行）
- [ ] 固定证据（三层断言 + 完整事故包）
- [ ] 固定验收规则（独立 verifier 复核）
- [ ] 固定失败分级（8 类故障自动分类）

**只有黄金路径全部达到上述验证标准，才扩展更多 case。否则继续扩 case 只会增加噪音。**

## 7. 约束

### 7.1 跨层引用约束

- 实施轴**必须**通过 `_ref` 字段引用需求轴资产
- 需求轴**不得**引用实施轴产物
- candidate artifact 中必须包含 `test_set_ref` 和 `test_environment_ref`
- verifier 不得直接引用 runner 的执行过程，只能引用其产物

### 7.2 不可变性约束

- Manifest 项永不删除，只标记 `obsolete`
- Evidence 文件只写不改，通过 SHA-256 哈希验证完整性
- Run Manifest 一旦生成不可修改
- Bug 记录一旦创建不可修改
- 事故包只追加，不覆盖

### 7.3 执行边界约束

- `TESTSET` 中不得嵌入 `test_case_pack` 或 `script_pack`（由 `_validate_testset_execution_boundary()` 校验）
- 用例扩展（qualification expansion）属于运行时行为，不写入 SSOT
- 三层断言中 C 层（业务状态断言）按阶段要求执行（见 2.5 节），黄金路径 C 层缺失 → gate 不得返回 pass
- UI E2E 模式下检测到 API 直调 → 自动 fail

### 7.4 Gate 判定维度

Gate 评价必须公开以下维度及其权重：

| 维度 | 要求 | 说明 |
|------|------|------|
| 用例通过率 | 黄金路径 100%，其余 >= 95% | 基础质量 |
| 三层断言覆盖率 | 黄金路径 C 层 100% 覆盖 | 断言质量 |
| Bypass 违规数 | 必须 = 0 | 执行合规 |
| Verifier verdict | 必须 != fail | 独立验证 — **verifier = fail → Gate 必须 = fail，不可被 settlement 覆盖** |
| 事故包中 PRODUCT 类故障 | 必须 = 0（或已有 workaround） | 真实 bug |
| 环境一致性 | Run Manifest 完整 | 可复现性 |

> **Phase 1 Gate 结论上限：** Phase 1（黄金路径 MVP）仅支持 A+B 层断言，C 层标记 `C_MISSING`。此阶段 Gate 结论上限为 `provisional_pass`（临时通过），仅用于技术验证和框架搭建，**不作为质量决策依据**。
>
> **`provisional_pass` 展示格式要求：**
> - 必须在报告**标题行**出现，不可隐藏在正文中。格式：`⚠ GATE: PROVISIONAL_PASS (C-layer assertions not verified)`
> - 必须在报告顶部列出 `C_MISSING` 的 case 数量和占比：`C_MISSING: X/Y cases (Z%)`
> - 必须包含明确声明："此结论不作为质量决策依据，C 层业务状态断言尚未验证"
> - **受众限制：** 此结论仅供技术团队内部参考，不得向非技术利益相关者展示或作为发布依据。

### 7.5 追溯链扩展

- Bug bundle → TESTSET.test_units 的追溯链：从 bug 反推到对应的测试用例，再到原始需求，形成完整追溯闭环
- PRODUCT 类故障修复后，必须有对应的回归 case 加入 TESTSET

### 7.6 Verifier 独立性约束

- verifier 必须使用独立的认证上下文（不同 API token、不同浏览器 context）
- Run Manifest 中分别记录 `runner.account` 和 `verifier.account`
- verifier 的测试数据必须与 runner 隔离（不同账号、不同数据快照）
- verifier 不得复用 runner 的 connection pool

## 8. 对 ADR-047 的影响

ADR-047 Section 11.1 定义的 5 层技能结构需调整为本 ADR 的双轴 + 2 入口模型：

| 原 ADR-047 5 层 | 新归属 | 变化 |
|-----------------|--------|------|
| 设计层（7 个 plan/manifest/spec 技能） | **需求轴模块** | 不变，重命名，由 `qa.test-plan` 编排 |
| 生成层（2 个 spec-to-tests 技能） | **实施轴模块** | 并入 `scenario-spec-compile` + 状态机执行，由 `qa.test-run` 编排 |
| 执行层（2 个 execute 技能） | **实施轴模块** | 升级：增加状态机约束，由 `qa.test-run` 编排 |
| 结算层（3 个 settlement/gate 技能） | **实施轴模块** | 不变，由 `qa.test-run`（验证+结算阶段）编排 |
| 兼容层（1 个 render-testset-view） | **需求轴模块** | 不变 |
| — | **实施轴模块（新增）** | 10 个新模块，不注册为 Skill |
| — | **用户入口 Skill（新增）** | 2 个：`qa.test-plan` / `qa.test-run`（qa.test-report 合并入 qa.test-run） |
