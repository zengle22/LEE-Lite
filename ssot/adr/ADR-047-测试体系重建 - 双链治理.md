# ADR-047：测试体系重建——基于双链治理的 API / E2E 测试架构

> **SSOT ID**: ADR-047
> **Title**: 测试体系重建——基于双链治理的 API / E2E 测试架构，并集成 BMAD 能力层
> **Status**: Trial Approved (v1.4)
> **Version**: v1.4
> **Pilot Scope**: 选择一个真实 feat + prototype 跑通双链治理全流程
> **Effective Date**: 2026-04-10
> **Scope**: LL / LEE 项目及后续采用 SSOT 驱动研发的项目
> **Owner**: 测试架构 / 质量治理
> **Governance Kind**: TRANSFER / DELIVERABLE
> **Audience**: 产品、架构、后端、前端、测试、AI 实施代理

---

## 1. 背景

### 1.0 One-Sentence Summary

> **不是把 `feat2testset` 升级一下，而是把它拆成"feat/prototype 下游的测试编译链"，让执行器改吃 spec，让 gate 改吃 settlement。**

### 1.1 旧主链路径

当前测试体系的旧主路径为：

```
raw -> src -> epic -> feat -> testset -> impl/test exec
```

该体系在实际 AI 驱动研发中暴露出明显问题：

### 1.2 新主链路径（双链治理）

测试体系重构为两条平行主链，从 feat/prototype 派生出 test-design：

```
raw -> src -> epic -> feat
                  ├-> tech
                  ├-> prototype
                  └-> test-design
                        ├-> api-test-plan
                        ├-> api-coverage-manifest
                        ├-> api-test-spec
                        ├-> e2e-journey-plan
                        ├-> e2e-coverage-manifest
                        └-> e2e-journey-spec
```

**关键变化**：
- `testset` 不再是源点，降级为 compatibility view / aggregation layer
- `test-design` 作为并行分支，从 `feat` 和 `prototype` 双输入派生
- `api-test-spec` 和 `e2e-journey-spec` 是脚本可生成、可执行、可验收的最终工件
- `settlement` 结果与 `waiver` 决策共同进入 `release_gate_input`

### 1.3 测试对象粒度过粗

现有 testset 更像"测试建议集合"，不是可治理的覆盖合同。API 规则、边界、状态机、异常分支、权限、幂等等关键点缺少显式覆盖债务管理。E2E 也容易退化为只跑一两条简单主流程，而不是覆盖完整用户旅程。

### 1.4 通过判定偏 narrative，缺少结构化证据结算

AI 可以通过生成看起来合理的测试报告来制造"已完成"的表象。即使增加监督与完成标准，也仍可能存在"未真实执行 / 未真实断言 / 未完整覆盖"的情况。

### 1.5 测试验收锚点不清晰

API 测试和 E2E 测试的责任边界混杂。测试既没有明确锚定 feat，也没有明确锚定 prototype。导致测试结果无法成为可靠的功能验收依据。

### 1.6 现有通用测试生成 skill 无法直接解决治理问题

包括 BMAD 在内的测试 skill 更擅长"生成测试框架、测试文档、测试脚本、CI 配置"，但并不天然提供本项目所需的测试治理闭环：

- 覆盖范围冻结
- 验收锚点绑定
- manifest 账本管理
- evidence 结算
- release gate 接入

---

## 2. 问题陈述

本项目需要一套能够在 AI 驱动研发下稳定工作的测试体系，满足以下目标：

### 2.1 后端 API 测试

- 做完整功能覆盖
- 以 feat 为验收锚点
- 验证功能能力、业务规则、状态机、异常路径、边界、权限、幂等、副作用

### 2.2 前端 E2E 测试

- 走完所有关键用户旅程
- 以 prototype 为验收锚点
- 验证真实 UI/UX 交互、页面反馈、关键旅程闭环、前后端集成结果

### 2.3 测试设计、管理、实施分离

- 不能在执行时再自由决定测什么
- 测试覆盖范围必须前置冻结并进入 SSOT 管理

### 2.4 测试结果必须可治理

能回答"哪些已覆盖、哪些未覆盖、哪些已执行、哪些没有证据、哪些不可放行"，最终进入 release gate。

### 2.5 可复用外部能力，但治理权必须保留在本项目

- BMAD 可作为能力层集成
- 但不能成为测试治理的真理源

---

## 3. 决策

本 ADR 作出以下核心决策：

### 3.1 测试体系重构为"双链治理"

测试体系拆分为两条主链，以 `test-design` 为并行分支：

#### A. API 测试链

**目标**：完整功能覆盖

**验收锚点**：feat

**职责**：
- 覆盖后端功能能力
- 校验业务规则与输入输出
- 校验状态机与合法/非法状态迁移
- 校验边界、异常、权限、幂等、数据副作用

**不负责**：
- 页面布局与视觉结构
- 用户 UI 交互体验
- prototype 展示细节本身

#### B. E2E 测试链

**目标**：完整关键用户旅程覆盖

**验收锚点**：prototype

**职责**：
- 覆盖主旅程、分支旅程、异常旅程、回访旅程
- 验证真实浏览器中的交互、反馈、跳转、持久化、关键闭环
- 验证 prototype 对应的用户体验是否真实落地

**不负责**：
- 替代 API 全量规则测试
- 穷举后端所有异常与业务边界

### 3.2 测试设计权前移，执行阶段不得定义覆盖范围

执行阶段不再负责"决定测什么"，仅允许：
- 根据已冻结 spec 生成实现
- 执行测试
- 回填证据和结果

执行阶段**不得**：
- 自行删减覆盖范围
- 自行决定某项可不测
- 用 narrative 报告替代结构化覆盖结算
- 没有证据却宣称通过

### 3.3 双输入测试设计模型

`test-design` 作为 `feat` 和 `prototype` 下游的编译链源点：

```
feat + prototype + optional(tech/api) -> test-design orchestrator -> api chain / e2e chain
```

- API 链：以 `feat` 为 Anchor，从 `feat` 中冻结的功能契约派生
- E2E 链：以 `prototype` 为 Anchor，从 `prototype` 中冻结的用户旅程派生
- `tech/api` 为可选输入，用于补充技术细节

### 3.2 测试设计权前移，执行阶段不得定义覆盖范围

执行阶段不再负责"决定测什么"，仅允许：
- 根据已冻结 spec 生成实现
- 执行测试
- 回填证据和结果

执行阶段**不得**：
- 自行删减覆盖范围
- 自行决定某项可不测
- 用 narrative 报告替代结构化覆盖结算
- 没有证据却宣称通过

### 3.3 测试资产四层分离——严格边界定义

不再使用单一 testset 作为测试真理源。测试资产分为四层，每层有明确职责边界：

| 层级 | 资产类型 | 核心问题 | 不得包含 |
|------|----------|----------|----------|
| **Plan** | `api-test-plan` / `e2e-journey-plan` | 要测哪些对象/旅程、优先级是什么、裁剪了什么 | 具体测试断言、执行状态、证据引用 |
| **Manifest** | `api-coverage-manifest` / `e2e-coverage-manifest` | 每个 coverage item 当前处于什么状态，映射到哪些 case，证据在哪里 | 测试设计细节、断言逻辑、脚本内容 |
| **Spec** | `api-test-spec` / `e2e-journey-spec` | 单个 case 怎么测、断言什么、证据要求什么 | 执行结果、状态变更、放行决策 |
| **Settlement** | `api-settlement-report` / `e2e-settlement-report` | 本轮结算结果和放行建议 | 测试设计、脚本实现 |

**关键原则**：
- Plan 只定义范围，不定义如何测
- Manifest 只追踪状态，不定义设计
- Spec 只定义合同，不包含执行结果
- Settlement 只结算结果，不回溯设计

#### API 侧四层资产

| 层级 | 资产 | 职责 |
|------|------|------|
| Plan | `api-test-plan` | 声明本轮 API 测试的覆盖范围与优先级 |
| Manifest | `api-coverage-manifest` | API 测试覆盖总账本，是放行依据之一 |
| Spec | `api-test-spec` | 定义单个 API 场景的结构化测试合同 |
| Settlement | `api-settlement-report` | API 测试本轮结算结果与放行建议 |

#### E2E 侧四层资产

| 层级 | 资产 | 职责 |
|------|------|------|
| Plan | `e2e-journey-plan` | 声明需要覆盖的关键用户旅程 |
| Manifest | `e2e-coverage-manifest` | E2E 旅程覆盖总账本 |
| Spec | `e2e-journey-spec` | 定义单个 E2E 场景的结构化测试合同 |
| Settlement | `e2e-settlement-report` | E2E 测试本轮结算结果与放行建议 |

### 3.4 API 测试锚定 feat，E2E 测试锚定 prototype

#### API -> feat

API 测试必须由 feat 中冻结的功能契约派生。API 测试覆盖不是对齐 prose，而是对齐 feat 中的：
- 功能对象行为定义
- 输入输出规则约束
- 状态机验收标准

#### E2E -> prototype

E2E 测试必须由 prototype 中冻结的用户旅程与交互契约派生。E2E 测试覆盖不是对齐静态页面截图，而是对齐 prototype 中的：
- 页面入口
- 关键用户动作序列
- 页面反馈
- 页面跳转与状态保留
- 关键展示结构
- 主 / 分支 / 异常 / 回访旅程

### 3.5 BMAD 仅作为"能力层"，不作为"治理层"

BMAD 的测试相关 skill 可集成，但其角色限定为：
- 测试框架初始化
- 测试设计草稿生成
- ATDD / E2E 场景辅助生成
- 测试脚本生成
- CI/CD 配置生成

BMAD **不得**：
- 定义本项目测试覆盖边界
- 替代本项目 SSOT 作为测试真理源
- 修改 API / E2E 的验收锚点与职责边界
- 独立决定 release gate

### 3.6 测试验收与放行以 manifest + evidence 为准

最终放行不再依据 narrative 测试报告，而依据：
- coverage manifest
- execution evidence
- gap report
- waiver decision

测试报告应退化为**覆盖结算单**，而不是"测试故事"。

### 3.7 testset 的去化策略

`testset` 是本项目测试体系重构的核心改造对象：

**testset 不再是源点真理**：
- `testset` 停止承担 "test coverage definition + test execution entry + acceptance basis" 三位一体角色
- `testset` 降级为 compatibility view / aggregation layer，用于向后兼容现有系统

**新的结构**：
- `feat` + `prototype` + `optional(tech/api)` -> `test-design orchestrator` -> `api chain` / `e2e chain`
- `api-test-plan` / `e2e-journey-plan` 作为覆盖率定义的源点
- `api-test-spec` / `e2e-journey-spec` 作为执行的最小契约
- `api-settlement-report` / `e2e-settlement-report` + `waiver` 作为验收依据

**旧的 testset 流程**（已废弃）：
```
feat -> testset -> test exec -> narrative report -> gate
```

**新的双链治理流程**（当前采纳）：
```
feat -> test-design -> api-test-spec -> execution -> evidence -> api-settlement
prototype -> test-design -> e2e-journey-spec -> execution -> evidence -> e2e-settlement
api-settlement + e2e-settlement + waiver -> release_gate_input -> gate
```

### 3.8 编译链设计

测试编译链是将设计转化为可执行脚本的核心机制，分为两个步骤：

#### Step 1: feat2testplan / prototype2testplan - 设计骨架生成

输入：`feat` / `prototype` + `tech/api`（可选）

输出：
- `api-test-plan` + `api-coverage-manifest`（草稿）
- `e2e-journey-plan` + `e2e-coverage-manifest`（草稿）

职责：
- 从feat/prototype冻结的契约中派生覆盖范围
- 初始化 coverage manifest 的初始状态账本
- 定义测试优先级与裁剪记录

#### Step 2: testplan2spec - 测试合同生成

输入：`api-test-plan` / `e2e-journey-plan` + `api-coverage-manifest` / `e2e-coverage-manifest`

输出：
- `api-test-spec`（单个 case 的结构化测试合同）
- `e2e-journey-spec`（单个 journey 的结构化测试合同）

职责：
- 将 plan/manifest 的抽象覆盖项编译为具体可执行的 spec
- 定义请求/断言/证据要求等执行细节
- 使测试具备"script-generatable / executable / acceptable"特性

**编译链价值**：
- 编译过程可追溯：每个 spec 都能回溯到 plan -> manifest -> feat/prototype
- 编译产物可执行：spec 是生成测试脚本的最小确定性契约
- 编译链可验证：每一步输出都可被机器校验格式与完整性

## 4. 详细设计

### 4.1 API 测试链设计

#### 4.1.1 API 测试目标

API 测试负责验证：
- 功能能力是否存在且与 feat 一致
- 输入输出是否符合约定
- 业务规则是否正确
- 状态机与状态迁移是否正确
- 异常路径与业务拒绝是否正确
- 权限控制是否正确
- 幂等/重复请求/并发行为是否符合预期
- 数据副作用是否正确

#### 4.1.2 API 测试的覆盖维度矩阵

每个 API capability / endpoint / action 必须显式评估以下覆盖维度：

| 维度 | 覆盖内容 |
|------|----------|
| **正常路径** | 合法输入、正确响应、正确副作用 |
| **参数校验** | 必填缺失、类型错误、格式错误、非法枚举、越界值 |
| **边界值** | 最小值/最大值、空集合/空字符串、时间边界、列表边界、长度边界 |
| **状态约束** | 合法状态迁移、非法状态迁移、重复提交、已完成后二次操作 |
| **权限与身份** | 未登录、无权限、跨用户访问、不同角色差异 |
| **异常路径** | 资源不存在、冲突、上游依赖失败、业务拒绝、可控内部异常 |
| **幂等/重试/并发** | 重复请求、重试请求、并发修改/并发提交 |
| **数据副作用** | DB 正确写入/更新/删除、关联对象同步变化、无额外脏数据、状态一致性保持 |

#### 4.1.3 API 覆盖裁剪规则表

**并非所有 capability 都必须完整走满所有维度，但必须显式裁剪**。裁剪规则如下：

| 优先级 | 必测维度 | 选测维度 | 裁剪要求 |
|--------|----------|----------|----------|
| **P0** | 正常路径、参数校验 (key)、状态约束、异常路径 (key)、数据副作用 | 边界值 (edge cases)、幂等/并发 | 任何裁剪必须有 `cut_reason`、`approver`、`source_ref` |
| **P1** | 正常路径、参数校验 (key)、异常路径 (key) | 边界值、状态约束、权限、幂等/并发 | 任何裁剪必须有 `cut_reason`、`source_ref` |
| **P2** | 正常路径 | 其他所有维度按风险选测 | 选测维度可省略但需记录 |

**维度覆盖数量说明**：

| 误解 | 正确理解 |
|------|----------|
| "P0 capability 必须有至少 5 项覆盖" | P0 的"必须覆盖"指的是**适用的维度必须覆盖**，不是固定的项数阈值 |
| "每个 capability 必须测试所有 8 个维度" | 8 维度是参考矩阵，capability 只覆盖**业务范围内适用的维度** |
| "少于 5 项就是覆盖不足" | 覆盖项数由业务范围决定，例如"日志记录"能力只需要覆盖正常路径+数据副作用 |

**示例**：

| Capability | 业务范围 | 适用维度 | 覆盖项数 |
|------------|----------|----------|----------|
| SEC-AUDIT-LOG | 安全事件记录到审计日志 | 正常路径、数据副作用 | 2 项 ✅ |
| SEC-REJECT-INVALID | 非法 payload 拦截 | 正常路径、异常路径、数据副作用 | 3 项 ✅ |

**默认规则**：没有明确裁剪记录的维度，默认视为 required。

**裁剪记录格式**（API / E2E 统一）：

```yaml
cut_record:
  cut_target: "维度名称 | journey_type"
  cut_reason: "裁剪原因说明"
  source_ref: "feat.xxx.rationale | proto.xxx.page-type"
  approver: "审批人姓名/角色"
  approved_at: "ISO8601 时间戳"
```

#### 4.1.4 Feat 最小可测试要求

**前置约束**：不满足最小可测试要求的 feat，不得进入 API 测试设计。

**最小可测试要求**至少包括：

| 要求 | 说明 |
|------|------|
| capability 明确 | 每个功能对象必须有明确定义的能力列表 |
| 输入输出明确 | 每个 capability 必须有明确的输入/输出契约 |
| 关键业务规则明确 | 核心业务规则必须有明确的验收条件 |
| 状态迁移明确 | 涉及状态变化的必须有明确的状态机定义 |
| 成功/失败验收标准明确 | 每个 capability 必须有明确的 success/failure 判定标准 |

#### 4.1.5 API 测试资产定义

##### A. `api-test-plan`

作用：声明本轮 / 本模块 API 测试的覆盖范围与优先级

**核心字段**：只回答"要测哪些对象/旅程、优先级是什么、裁剪了什么"

```yaml
api_test_plan:
  feature_id: feat.xxx
  source_feat_refs:
    - feat.xxx.section.1
    - feat.xxx.section.2
  api_objects:
    - name: training_plan
      capabilities:
        - create_plan
        - get_plan
        - update_plan
    - name: feedback
      capabilities:
        - submit_feedback
  priorities:
    p0:
      - create_plan
      - submit_feedback
    p1:
      - get_plan
      - update_plan
  cut_records:
    - capability: update_plan
      dimension: 幂等/重试/并发
      cut_reason: "计划更新为覆盖式，无并发冲突场景"
      source_ref: feat.training-plan.rationale.no-concurrency
```

##### B. `api-coverage-manifest`

作用：API 测试覆盖总账本，是放行依据之一

**核心字段**：只回答"每个 coverage item 当前处于什么状态，映射到哪些 case，证据在哪里"

```yaml
api_coverage_manifest:
  items:
    - coverage_id: api.plan.create.happy
      feature_id: feat.training-plan.create
      capability: create_plan
      endpoint: POST /api/plans
      scenario_type: happy_path
      priority: P0
      source_feat_ref: feat.training-plan.acceptance.create-success
      dimensions_covered:
        - 正常路径
        - 数据副作用
      mapped_case_ids:
        - api_case.plan.create.happy
      # 分层状态字段（避免单字段语义过载）
      lifecycle_status: designed  # 生命周期阶段
      mapping_status: mapped      # 映射状态：unmapped | mapped | superseded
```

### 4.2 E2E 测试链设计

#### 4.2.1 E2E 测试目标

E2E 测试负责验证：
- prototype 中定义的关键用户旅程是否被真实实现
- 用户在真实浏览器中是否可完成关键操作
- 页面交互、反馈、跳转、持久化是否符合 prototype
- 主闭环、关键分支、关键异常处理是否存在
- 前后端在用户视角下是否真实打通

#### 4.2.2 E2E 用户旅程分类

每个 prototype 模块至少识别以下旅程类型：

| 类型 | 说明 |
|------|------|
| **主旅程** | 主业务闭环路径 |
| **分支旅程** | 主路径中不同选择、不同输入、不同页面走法 |
| **异常旅程** | 表单错误、接口失败、状态冲突、空态等 |
| **回访旅程** | 用户再次进入系统的路径 |
| **状态型旅程** | 草稿态、已提交态、已完成态、不可编辑态等 |

#### 4.2.3 E2E 旅程识别与最小枚举规则

**每个 prototype 必须识别的最小旅程集合**：

| 触发条件 | 必须识别的旅程 | 优先级 |
|----------|----------------|--------|
| 每个 prototype 页面流 | 至少 1 条主旅程 | P0 |
| 每个表单页 | 至少 1 条校验失败异常旅程 | P0 |
| 每个有状态差异的页面 | 至少 1 条状态型旅程 | P1 |
| 每个有回访入口的闭环 | 至少 1 条 revisit 旅程 | P1 |
| 每个主旅程 | 至少 1 条关键分支旅程 | P1 |
| 每个网络请求点 | 至少 1 条网络失败异常旅程 | P1 |

**默认规则**：没有显式声明裁剪的，默认适用上述最小枚举规则。

**裁剪记录格式**（与 API 侧统一）：

```yaml
journey_cut_records:
  - cut_target: journey_type.exception
    cut_reason: "页面为纯展示页，无用户输入"
    source_ref: proto.xxx.page-type.readonly
    approver: "product-lead-name"
    approved_at: "2026-04-10T10:00:00Z"
```

#### 4.2.4 E2E 断言维度

每条 journey 至少检查：

| 维度 | 检查内容 |
|------|----------|
| **UI 状态** | 页面显示、关键字段、组件状态、禁用/启用、loading/empty/error |
| **交互反馈** | 成功提示、错误提示、页面反馈时机 |
| **Network 行为** | 请求是否真实发出、请求数量/时机是否正确、响应成功/失败是否正确处理 |
| **持久化与状态延续** | reload 后仍保持、重新进入后仍正确、后端数据与 UI 一致 |
| **Anti-false-pass 检查** | 页面成功提示不能替代真实成功、local state 更新不能替代服务端持久化、页面跳转不能替代业务完成、console error 不得被忽略 |

#### 4.2.5 E2E 测试资产定义

##### A. `e2e-journey-plan`

**前置约束**：不满足最小可测试要求的 prototype，不得进入 E2E 设计。

**最小可测试要求**至少包括：

| 要求 | 说明 |
|------|------|
| 页面入口明确 | 每个旅程必须有明确的 entry_point |
| 主旅程至少一条 | prototype 必须包含至少一条完整主旅程 |

| 关键页面反馈 | 成功/失败/加载状态必须有明确反馈 |

## 5. 测试设计流程

### 5.1 API 测试设计流程

```
feat
  -> api-test-plan           # 定义范围与优先级
  -> api-coverage-manifest   # 初始化账本
  -> api-test-spec           # 定义测试合同
  -> script generation       # 生成脚本
  -> execution               # 执行测试
  -> evidence                # 收集证据
  -> coverage settlement     # 结算覆盖
```

**设计原则**：
- 先识别 capability，再识别 scenario
- 先冻结 coverage item，再生成 case
- coverage item 必须能够追溯到 feat
- 每个 coverage item 必须有明确 scenario_type 与 evidence requirement

### 5.2 E2E 测试设计流程

```
prototype
  -> e2e-journey-plan        # 定义旅程范围
  -> e2e-coverage-manifest   # 初始化账本
  -> e2e-journey-spec        # 定义测试合同
  -> script generation       # 生成脚本
  -> execution               # 执行测试
  -> evidence                # 收集证据
  -> coverage settlement     # 结算覆盖
```

**设计原则**：
- 先识别用户旅程，再识别分支与异常
- 先冻结 journey 覆盖，再生成脚本
- 每条旅程必须能够追溯到 prototype
- E2E 必须包含 anti-false-pass checks
- prototype 必须满足最小可测试要求

## 6. 测试实施流程

### 6.1 执行阶段允许的动作

执行阶段只允许：
- 读取已冻结的 `api-test-spec` / `e2e-journey-spec`
- 根据 spec 生成测试脚本 / 补齐执行实现
- 真实执行
- 收集 evidence
- 回填 manifest 与 settlement report

### 6.2 执行阶段禁止的动作

执行器不得：
- 自行删减 manifest item
- 自行改变 case scope
- 以截图替代真实断言
- 以页面展示替代业务成功
- 以 narrative 报告替代 evidence ledger
- 在 evidence 不全时标记 passed

### 6.3 Evidence 要求

#### API evidence 至少包含
- request snapshot
- response snapshot
- assertion results
- side-effect assertion / db assertion
- execution log

#### E2E evidence 至少包含
- browser trace
- screenshot / video（视配置）
- network log
- DOM assertion result
- persistence assertion
- console error check result

### 6.4 新执行流程（双链执行）

#### API 链执行流程
```
api-test-spec -> api generation -> api execution -> api evidence -> api settlement
```

**说明**：
- `api generation`：根据 spec 生成 pytest/unittest 等脚本
- `api execution`：执行脚本并记录真实调用与响应
- `api evidence`：收集 request/response 断言结果、DB 副作用等证据
- `api settlement`：结算 coverage item 的最终状态

#### E2E 链执行流程
```
e2e-journey-spec -> e2e generation -> e2e execution -> e2e evidence -> e2e settlement
```

**说明**：
- `e2e generation`：根据 spec 生成 Playwright/Cypress 脚本
- `e2e execution`：执行浏览器自动化并记录交互旅程
- `e2e evidence`：收集 trace、screenshot、network log、DOM assertion 等证据
- `e2e settlement`：结算 journey coverage 的最终状态

#### Release Gate 输入流程
```
api settlement + e2e settlement + waiver -> release_gate_input -> release gate
```

**说明**：
- `release_gate_input`：由 gate evaluator 读取 settlement + waiver 后生成的 YAML 文件
- `release gate`：CI/CD job 读取 `release_gate_input.yaml` 执行 pass/fail 判定
- `waiver`：对 fail/blocked/uncovered 项的审批豁免记录

## 9. 放行规则

### 9.1 API 放行规则

以下条件必须满足：
- 所有 P0 / P1 capability 已进入 `api-coverage-manifest`
- 所有 required coverage item 均已映射 case
- 所有 required case 均已执行
- 所有 executed case 均有 evidence
- fail / blocked / uncovered 项已显式列出
- 未审批 waiver 的 fail / blocked / uncovered 不得放行

### 9.2 E2E 放行规则

以下条件必须满足：
- 所有关键主旅程已进入 `e2e-coverage-manifest`
- 所有 P0 主旅程必须执行
- 所有关键异常旅程至少覆盖一轮
- 所有 executed case 均有真实浏览器 evidence
- 未覆盖旅程必须显式列出
- 未审批 waiver 的 P0 未覆盖项不得放行

### 9.3 通用规则

- 无 evidence 不算 executed
- executed 不等于 passed
- passed 不等于 releasable
- release gate 以 settlement report + waiver decision 为准

### 9.4 机器可执行的 Gate 规则

将放行规则编译为机器可执行字段：

```yaml
gate_rules:
  api:
    require_all_p0_mapped: true
    require_all_p1_mapped: true
    require_all_p0_executed: true
    require_all_p1_executed: true
    require_evidence_for_all_executed: true
    fail_if_unwaived_failed_count_gt: 0
    fail_if_unwaived_blocked_count_gt: 0
    fail_if_unwaived_p0_uncovered_count_gt: 0

  e2e:
    require_all_p0_main_journeys_mapped: true
    require_all_p0_main_journeys_executed: true
    require_min_exception_journey_coverage: 1
    require_evidence_for_all_executed: true
    fail_if_unwaived_failed_count_gt: 0
    fail_if_unwaived_blocked_count_gt: 0
    fail_if_unwaived_p0_uncovered_count_gt: 0
```

### 9.5 Gate Evaluator 与 CI 消费关系

**消费链**：

```
manifest + settlement + waiver
    -> gate evaluator (生成)
    -> release_gate_input.yaml
    -> CI/CD job (消费)
    -> pass/fail decision
```

**关键规则**：

| 角色 | 职责 | 不得 |
|------|------|------|
| **Gate Evaluator** | 读取 manifest/settlement/waiver，生成 `release_gate_input.yaml` | 直接决定放行，只生成输入 |
| **CI/CD Job** | 读取 `release_gate_input.yaml`，执行 pass/fail | 直接解释 manifest，不重写 gate 逻辑 |
| **Human Approver** | 审批 waiver，确认 final_decision | 绕过 gate 规则直接放行 |

**CI 读取的字段**：

```yaml
# CI/CD job 只读取此文件
.artifacts/tests/settlement/release-gate-input.yaml

release_gate_input:
  api:
    status: pass | conditional_pass | fail
    uncovered_count: number
    failed_count: number
    blocked_count: number
    waiver_refs: [...]
  e2e:
    status: pass | conditional_pass | fail
    uncovered_count: number
    failed_count: number
    blocked_count: number
  final_decision: release | conditional_release | block
```

**Rationale**：

- 避免 CI 实现层自己重写一套解释逻辑
- gate 逻辑变更只需改 gate evaluator，不需改 CI 配置
- `release_gate_input.yaml` 作为审计 trace 的固定锚点

## 11. 迁移策略

采用渐进式迁移，而非一次性推翻。

### 11.1 Skill 层级结构

测试体系的技能按职责分为五层：

#### 设计层 Skills
- `feat-to-api-test-plan`：从 feat 派生 API test plan 与 coverage manifest
- `feat-to-e2e-journey-plan`：从 feat 派生 E2E journey plan 与 coverage manifest（如适用）
- `prototype-to-e2e-journey-plan`：从 prototype 派生 E2E journey plan 与 coverage manifest
- `api-plan-to-manifest`：从 plan 编译 coverage manifest
- `e2e-plan-to-manifest`：从 plan 编译 coverage manifest
- `api-manifest-to-spec`：从 manifest 编译 spec
- `e2e-manifest-to-spec`：从 manifest 编译 spec

#### 生成层 Skills
- `api-spec-to-tests`：从 `api-test-spec` 生成 pytest/unittest 脚本
- `e2e-spec-to-tests`：从 `e2e-journey-spec` 生成 Playwright/Cypress 脚本

#### 执行层 Skills
- `execute-api-tests`：执行 API 测试并返回结果
- `execute-e2e-tests`：执行 E2E 测试并返回结果

#### 结算层 Skills
- `build-api-settlement`：从 evidence 构建 API settlement report
- `build-e2e-settlement`：从 evidence 构建 E2E settlement report
- `build-release-gate-input`：整合 settlement + waiver 生成 `release_gate_input.yaml`

#### 兼容层 Skills
- `render-testset-view`：从 plan/manifest/spec/settlement 聚合渲染旧 testset 视图，供兼容系统读取

### 11.2 迁移策略

### Phase 1：确立双链治理骨架

**目标**：
- 建立 API / E2E 双链边界定义
- 定义 plan / manifest / spec / settlement 四层资产

**优先接入**：

- `/bmad-testarch-ci`

**接入**：
- `/bmad-qa-generate-e2e-tests`

- 必须由 `e2e-journey-spec` 驱动

**要求**：

## 12. 影响分析

### 12.1 正面影响

| 影响 | 说明 |
|------|------|
| **测试边界清晰** | API 与 E2E 不再混杂 |
| **验收锚点清晰** | API -> feat，E2E -> prototype |
| **可治理性显著增强** | 能明确回答哪些已覆盖、哪些未覆盖、哪些不能放行 |
| **可复用 BMAD，减少重复造轮子** | 充分利用现成 skill 的生产能力 |
| **降低 AI 偷懒空间** | 覆盖范围前置冻结，evidence 必填，manifest 驱动 gate |
| **四层资产分离** | plan/manifest/spec/settlement 职责清晰，避免互相污染 |

### 12.2 负面影响 / 成本

| 影响 | 说明 |
|------|------|
| **测试资产数量增加** | 比单一 testset 更重 |
| **设计前置工作增多** | 必须先出 plan / manifest / spec |
| **执行链更严格** | 早期可能觉得"更慢"，但换来可控性与可信度 |
| **BMAD 不能无脑整包复用** | 需要做适配与边界约束 |

## 13. 不采纳的方案

### 方案 A：继续沿用单一 testset，强化提示词与监督

**不采纳原因**：
- 不能从结构上约束覆盖范围
- 依然会退化为少量 happy path + narrative completion

### 方案 B：让 BMAD 整套接管测试体系

**不采纳原因**：
- BMAD 解决的是测试生产，不是本项目的测试治理
- 无法保证 feat / prototype 锚点与本项目 release gate 一致

### 方案 C：只保留 E2E，不做完整 API 测试

**不采纳原因**：
- E2E 无法替代 API 的规则、边界、状态机、异常路径覆盖
- 会导致"用户表面流程可跑，但系统规则不完整"

### 方案 D：引入 ATDD 作为第三条独立主链

**不采纳原因**：
- 容易重新混淆测试边界
- 当前阶段应收敛，不应增加新的主治理维度

## 15. 附录 A：Manifest 状态机与统计口径

### A.1 分层状态字段定义

为避免单字段语义过载，manifest item 采用分层状态字段：

| 字段 | 类型 | 取值 | 说明 |
|------|------|------|------|
| `lifecycle_status` | string | `drafted\|designed\|generated\|executable\|executed\|passed\|failed\|blocked\|waived` | 生命周期阶段 |
| `mapping_status` | string | `unmapped\|mapped\|superseded` | case 映射状态 |
| `evidence_status` | string | `missing\|partial\|complete` | 证据完整性 |
| `waiver_status` | string | `none\|pending\|approved\|rejected` | 豁免审批状态 |
| `obsolete` | boolean | `true\|false` | 是否已失效（上游变更导致） |

**状态组合示例**：

```yaml
# 正常通过
lifecycle_status: passed
mapping_status: mapped
evidence_status: complete
waiver_status: none
obsolete: false

# 失败待审批
lifecycle_status: failed
mapping_status: mapped
evidence_status: complete
waiver_status: pending
obsolete: false

# 已失效（上游变更）
lifecycle_status: designed
mapping_status: superseded
evidence_status: N/A
waiver_status: none
obsolete: true
```

### A.2 生命周期状态流转

```
drafted -> designed -> generated -> executable -> executed -> {passed|failed|blocked}
                                                      blocked -> waived
                                                      failed  -> waived (需审批)
                                                      any     -> obsolete (上游变更)
```

### A.3 扩展追踪字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `rerun_count` | integer | 重跑次数，初始为 0 |
| `last_run_id` | string | 最后一次执行 ID |
| `superseded_by` | string | 被哪个新的 coverage_id / case_id 替代 |

### A.4 Gate 统计口径定义

**Denominator（分母）统计规则**：

| 统计项 | 计入分母条件 | 排除条件 |
|--------|--------------|----------|
| total_coverage_items | 所有 manifest items | `obsolete=true` 的 items |
| required_items | priority=P0 或 P1 的 items | `obsolete=true` 或 `waiver_status=approved` |
| executed_items | `lifecycle_status` ∈ {executed, passed, failed} | `obsolete=true` 的 items |
| passed_items | `lifecycle_status=passed` 且 `evidence_status=complete` | - |
| uncovered_items | `mapping_status=unmapped` | `obsolete=true` 的 items |
| failed_items | `lifecycle_status=failed` 且 `waiver_status≠approved` | `obsolete=true` 的 items |
| blocked_items | `lifecycle_status=blocked` 且 `waiver_status≠approved` | `obsolete=true` 的 items |

**Exception Journey  coverage 统计单位**：

- `require_min_exception_journey_coverage: 1` 表示**全局至少 1 条**异常旅程
- 如需每个主旅程对应 1 条异常旅程，应写为 `require_exception_journey_per_main_journey: true`

**Waiver 处理规则**：

- `waiver_status=pending` 的 items 仍计入 failed/blocked/uncovered
- `waiver_status=approved` 的 items 从分母排除
- `waiver_status=rejected` 的 items 必须修复后才能放行

## 18. 一句话结论

本 ADR 采纳的核心路线是：

> **以"双链治理"重建测试体系：API 测试完整覆盖 feat 定义的功能契约，E2E 测试完整覆盖 prototype 定义的关键用户旅程；BMAD 作为能力层集成，但测试覆盖定义、验收锚点、manifest 账本与 release gate 仍由本项目 SSOT 体系掌控。**

**v1.1 关键增强**：四层资产严格分离、覆盖裁剪规则、旅程枚举规则、机器可执行 gate、扩展状态机、API 脚本生成契约。

**v1.2 关键增强**：分层状态字段、Gate 统计口径、Feat 最小可测试要求、裁剪 schema 统一、CI 消费关系明确。

**v1.4 关键增强**：双链治理流程图、testset 去化策略、编译链设计、Skill 层级结构、三个迁移连接点。

**当前状态**：Trial Approved — 可进入试点实施。

## 19. 迁移连接点

采用渐进式迁移，而非一次性推翻。关键改造涉及三个核心连接点：

### Connection 1：入口改造 - `feat2testset` -> `feat/prototype -> test-design`

**旧方式**：
```
feat -> feat2testset -> testset -> test exec
```

**新方式**：
```
feat + prototype -> feat2testdesign -> test-design -> api-test-spec / e2e-journey-spec -> test exec
```

**改造要点**：
- 保留 `feat2testset` 作为兼容 wrapper
- 该 wrapper 内部调用 `feat2testdesign`，然后渲染旧 testset 视图
- 新流程直接输出 plan/manifest/spec 到新目录结构

### Connection 2：执行入口改造 - test exec 改吃 spec

**旧方式**：
```
testset -> test exec -> narrative report
```

**新方式**：
```
api-test-spec -> api generation -> api execution -> api evidence -> api settlement
e2e-journey-spec -> e2e generation -> e2e execution -> e2e evidence -> e2e settlement
```

**改造要点**：
- `test exec` 不再读取 testset
- 只消费 `api-test-spec` 或 `e2e-journey-spec`
- Spec 是生成脚本的最小确定性契约

### Connection 3：验收改造 - gate 改吃 settlement + waiver

**旧方式**：
```
narrative report -> release gate
```

**新方式**：
```
api settlement + e2e settlement + waiver -> release_gate_input -> release gate
```

**改造要点**：
- `release gate` 不再解析 narrative 报告
- 只读取 `release_gate_input.yaml`
- Admission decision 由 `api_status + e2e_status + uncovered_count + failed_count + waiver_refs` 共同决定
