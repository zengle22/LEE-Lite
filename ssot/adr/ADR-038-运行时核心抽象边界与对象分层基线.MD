# ADR-038：运行时核心抽象边界与对象分层基线

* 状态：Accepted
* 日期：2026-04-02
* 决策者：Le Zeng
* 适用范围：LEE Lite（LL）v0.x 全部 runtime、governed skill、workflow authoring、CLI control plane 与治理对象建模
* 相关 ADR：ADR-001、ADR-003、ADR-004、ADR-005、ADR-006、ADR-009、ADR-018、ADR-020、ADR-037

---

# 1. 背景

随着 LL 主链逐步从概念验证走向可持续运行，仓库里已经分别冻结了几类关键边界：

* ADR-018 冻结了 `job runner`、`job state machine` 与 execution loop；
* ADR-020 冻结了 `skill / adapter / CLI carrier` 的边界；
* ADR-037 开始把 `failure_case / diagnosis / remediation` 这一类治理对象正式资产化；
* `cli/ll.py`、`cli/lib/protocol.py`、`cli/lib/job_queue.py`、`cli/lib/skill_invoker.py` 已经把 command、job、loop、dispatch、trace 等运行时接口部分实现为正式协议；
* `skills/*/SKILL.md`、`ll.contract.yaml`、`ll.lifecycle.yaml`、`agents/*.md` 已经把 governed workflow skill 的静态 authority 基本固定下来；
* `.workflow/`、`artifacts/jobs/`、`artifacts/active/gates/`、`artifacts/evidence/` 已经承载了大量真实运行态对象。

但即使这样，仓库里仍持续出现一类跨层混淆：

* 把 `skill` 当成“任何能跑的东西”；
* 把 `workflow` 当成“某个 agent 自己的一轮输出”；
* 把 `tool` 退化成“脚本调用”而没有权限与证据语义；
* 把 `task` 继续理解成 `impl-task.md` 这种文件，而不是一级运行对象；
* 把 `.workflow/` 中的 session/run residue 与 `artifacts/` 中的正式对象混成一个层次；
* 把 `command`、`skill`、`CLI carrier`、`tooling CLI`、`control CLI` 互相冒充。

当前系统已经不能再靠口头解释维持这些边界。

因此，需要一份新的 Runtime 架构 ADR，统一回答：

## **在 LL v0.x 中，Command、Skill、Tool、Agent、Task、Workflow、Session、Artifact、Evidence、Gate 分别是什么，它们各自落在哪一层，谁拥有 authority，谁只是 carrier。**

---

# 2. 问题陈述

当前真正要解决的不是“再补一个术语表”，而是以下结构性问题。

## 2.1 Skill 被迫承载过多运行态逻辑

如果不明确 `skill` 只负责能力模板与治理合同，而把 loop、claim、dispatch、resume、queue ownership 都塞进 skill，就会直接破坏 ADR-001、ADR-018、ADR-020 已经建立的边界。

## 2.2 Workflow 与 Agent 角色互相穿透

仓库里已经同时存在：

* `workflow.*` 路由标识；
* `agents/executor.md` 与 `agents/supervisor.md`；
* `.workflow/claude-code/*` 这类会话痕迹；
* `skill_invoker.py` 中按 `target_skill` 路由的执行入口。

如果不把 workflow 定义成“多步流程 authority”，把 agent 定义成“执行主体”，后续会继续把“谁来执行”和“流程定义是什么”混成一件事。

## 2.3 Tool 仍停留在 carrier 级，而不是治理对象级

当前仓库对工具层只有部分冻结：

* ADR-001 与 API contract 已区分 `tooling CLI` 与 `control CLI`；
* `cli/lib/protocol.py` 已要求 request/response schema、`request_id`、`actor_ref`、`trace`、`evidence_refs`；
* 但仓库还没有独立的 `tool object` authority。

如果不明确这一点，后续实现会一边说“新增 tool”，一边实际落成：

* repo command
* skill 内脚本
* 临时 helper
* 控制面动作

最终仍然没有统一的权限和证据语义。

## 2.4 Task 仍被文件语义绑架

当前仓库中同时存在：

* `artifacts/jobs/*.json` 这种真实可 claim / resume / retry / audit 的运行对象；
* `impl-task.md` 这种 IMPL 包内的执行说明文档；
* `artifacts/task/` 这个预留目录。

如果不明确 `task` 是运行单元而不是 markdown 文件，后续所有恢复、审计、ownership、retry 都会继续错绑到文档层。

## 2.5 Session 与 Artifact 的边界不清

当前仓库已经同时存在：

* `.workflow/runs/*`
* `.workflow/claude-code/*`
* `.workflow/workspace/*`
* `artifacts/active/runner/contexts/*`
* `artifacts/active/gates/*`
* `artifacts/evidence/*`

如果不明确：

* 哪些是 runtime session residue；
* 哪些是正式可引用 artifact；
* 哪些只是 session 的投影快照；

那么仓库会继续把长期 authority 与短期运行态搅在一起。

因此，需要在架构层冻结一套统一抽象边界。

---

# 3. 决策

## 3.1 决策摘要

LL v0.x 正式采用以下 Runtime 抽象分层：

1. `Command` 是显式触发入口，不是能力本体。
2. `Skill` 是一类问题的 governed capability template，不持有长期运行态。
3. `Tool` 是原子执行能力，必须带 request/response schema、permission scope 与 evidence semantics。
4. `Agent` 是拥有角色、上下文边界、允许工具池与执行权限的主体。
5. `Workflow` 是多步流程 authority，不等于 agent，也不等于一次 command。
6. `Task` 是可追踪、可恢复、可审计的运行单元；在当前仓库里，runtime task authority 暂时由 `job object family` 承载。
7. `Session` 是可持续运行、可 resume / handoff / compact 的长期上下文对象；其原始状态默认留在 `.workflow/` 等 runtime 区域。
8. `Artifact` 是正式持久化产物；`Evidence` 是用于支撑结论、Gate、决策的专用 artifact。
9. `Gate` 是对状态推进进行约束的治理边界；只有 gate 才拥有“能否推进”的 authority。

同时正式冻结：

## **新能力必须先判定自己属于哪个抽象，再选择 carrier；不得再从 carrier 倒推抽象。**

也就是说：

* 不能因为先写了一个 `cli` 命令，就宣称实现了 skill；
* 不能因为先写了一个 `impl-task.md`，就宣称已经产生了 runtime task；
* 不能因为先有 `scripts/*.py`，就把它们直接叫 tool；
* 不能因为某个 agent 在对话里走了多步，就把这轮对话直接当 workflow authority。

---

## 3.2 先分 authority，再分 carrier

本 ADR 采用两层判断法：

### A. Authority Layer

回答“这个对象在架构上是什么”。

例如：

* governed capability template
* atomic execution capability
* running work unit
* decision boundary
* durable output

### B. Carrier Layer

回答“这个对象当前由什么文件、目录、CLI surface 或 runtime path 承载”。

例如：

* `skills/<...>/SKILL.md`
* `cli/commands/*`
* `artifacts/jobs/*.json`
* `.workflow/runs/*`
* `artifacts/evidence/*`

本 ADR 的硬规则是：

## **carrier 不得反向篡改 authority。**

因此：

* `cli/commands/skill/command.py` 是 skill 的 carrier，不是 skill 本体；
* `scripts/*.py` 是 workflow/tooling 的 carrier，不是 workflow authority；
* `impl-task.md` 是 artifact carrier，不是 runtime task authority；
* `.workflow/claude-code/*` 是 session residue carrier，不是正式 artifact authority。

---

## 3.3 Skill vs Workflow 判定规则

### 判定优先级

当一个能力同时看起来像 Skill 和 Workflow 时，按下面顺序判断：

1. 先看它是否主要在回答“做什么能力”，而不是“如何推进状态”。
2. 再看它是否主要在回答“阶段怎么走、在哪些点停、谁来审、何时 handoff”。
3. 如果长期状态 ownership、stage transition、gate、retry、resume、claim 是核心语义，则 primary 倾向 Workflow。
4. 如果输入输出 contract、角色约束、能力复用、稳定 prompt shell 是核心语义，则 primary 倾向 Skill。
5. 如果两者都强，canonical bundle 可以同时承载，但必须显式写清 primary / secondary，且不能把 secondary 偷换成主 authority。

### 操作化判定表

| 观察到的主信号 | primary 倾向 | 典型解释 |
| --- | --- | --- |
| 主要目标是复用同类能力、输出契约稳定、角色约束稳定 | Skill | 能力模板是主语义，流程只是实现细节 |
| 主要目标是阶段推进、状态迁移、gate / handoff / review 编排 | Workflow | 流程推进是主语义，模板只是承载方式 |
| 只有单轮或线性步骤，且不持有外部状态 ownership | Skill | 更像一个 governed capability shell |
| 需要 claim / retry / resume / lease / recovery | Workflow | 说明对象已经进入运行态编排层 |
| 主要价值在“同一能力可被多处调用” | Skill | 复用性优先于编排性 |
| 主要价值在“同一流程必须按固定顺序被推进” | Workflow | 编排性优先于复用性 |

### 示例表

| 场景 | primary | secondary | 说明 |
| --- | --- | --- | --- |
| 只有输入输出 contract 和角色 prompt 的单轮 governed capability | Skill | Workflow 不显式化 | 重点是能力模板 |
| 多阶段、带 gate、带 handoff、带状态迁移的链路 | Workflow | Skill 作为承载壳 | 重点是流程推进 |
| 既要稳定能力复用，又要固定阶段推进的 canonical bundle | 视核心语义而定 | 另一层必须显式声明 | 不能同时把两者都写成主语义 |
| `job runner` 这类需要 claim / lease / retry 的运行器 | Workflow | Skill 仅作用户入口壳时才可保留 | 运行态编排优先 |
| `gate human orchestrator` 这类先决策再推进的治理壳 | Workflow | Skill / Command 作为 carrier | 推进与决策优先 |

### 一句判定法

> 如果去掉阶段推进以后，这个对象仍然完整成立，primary 多半是 Skill；如果去掉阶段推进以后，这个对象就失去存在理由，primary 多半是 Workflow。

## 3.4 核心定义总表

| 概念 | 正式定义 | 当前仓库 authority | 当前主要 carrier | 明确不等于 |
| --- | --- | --- | --- | --- |
| Command | 人或控制层显式触发的入口契约 | 调用入口 | `ll <group> <action>`、installed skill adapter、request envelope | skill、tool、workflow |
| Skill | 针对一类问题的 governed capability template | `skills/...` canonical bundle | `SKILL.md`、`ll.contract.yaml`、`agents/`、`scripts/` | command、session、job |
| Tool | 原子执行能力，带 schema / 权限 / 证据语义 | 当前以 CLI interface family 形式部分冻结 | `tooling CLI`、`control CLI`、protocol request/response | skill、workflow、agent |
| Agent | 拥有角色、上下文、工具池、权限边界的执行主体 | role specification | `agents/executor.md`、`agents/supervisor.md`、human gate operator、installed adapter runtime | workflow、skill authority |
| Task | 可追踪、可恢复、可审计的运行单元 | 当前由 `job object family` 承载 | `artifacts/jobs/*.json`、`trace.run_ref`、execution outcomes | `impl-task.md`、普通 artifact |
| Workflow | 多步流程编排与状态推进定义 | governed workflow contract | `ll.contract.yaml`、`ll.lifecycle.yaml`、runtime scripts、`workflow.*` route key | agent、command |
| Session | 可持续运行、可 resume / handoff / compact 的长期上下文对象 | runtime context | `.workflow/`、runner contexts、conversation logs | artifact authority、workflow definition |
| Artifact | 正式持久化、可引用、可复用、可评审的产物 | repo durable object | `artifacts/`、`ssot/`、formal / report / handoff / package files | session residue |
| Evidence | 用于支撑结论、Gate、决策、恢复动作的专用 artifact | audit / execution / supervision proof | `artifacts/evidence/`、`evidence_refs`、audit finding bundles | 任意普通 artifact |
| Gate | 对状态推进进行约束并生成治理决定的门禁对象 | gate authority | `artifacts/active/gates/*`、decision / dispatch / pending / brief records | runner、skill、tool |

# 4. 各抽象的正式边界

## 4.1 Command

### 定义

`Command` 是一次显式调用入口。

它负责：

* 选择入口 surface；
* 接收 request envelope；
* 绑定 `request_id / actor_ref / trace / payload`；
* 决定把调用交给 skill、tooling CLI 或 control CLI。

当前仓库中的直接证据包括：

* `cli/ll.py` 中的 `artifact / registry / audit / gate / loop / job / rollout / skill / validate / evidence` 分组；
* `cli/lib/protocol.py` 中固定的 request/response envelope；
* installed skill adapter 对 canonical skill 的 operator-facing 触发。

### 规则

1. Command 是“怎么进来”，不是“能力是什么”。
2. 一个 command 可以路由到 skill，也可以路由到 control plane。
3. Command 不拥有长期生命周期；生命周期属于 session、task 或 workflow run。

### 不得混淆

以下都不应再被叫成“skill 本身”：

* `ll skill failure-capture`
* `ll loop run-execution`
* `ll gate dispatch`

它们首先是 command。

---

## 4.2 Skill

### 定义

`Skill` 是针对一类问题的 governed capability template。

它负责：

* 定义输入输出 contract；
* 定义角色分工；
* 定义 evidence obligations；
* 定义 lifecycle、freeze、revision 规则；
* 给执行主体提供稳定 prompt / workflow shell。

当前仓库中的 authority 已由 ADR-020 冻结：

* canonical governed skill bundle 在 `skills/...`；
* 核心文件为 `SKILL.md`、`ll.contract.yaml`、`ll.lifecycle.yaml`、`input/`、`output/`、`agents/`、`evidence/`；
* installed adapter 是运行副本，不是 authority。

### 规则

1. Skill 是静态 authority，不持有长期 queue / lease / claim / session residue。
2. Skill 可以调用 tooling CLI，但不能越权调用 control CLI 触发新的执行循环。
3. Skill 可以实现一个 workflow，也可以是 workflow 的用户入口壳，但 skill 不等于 command。

### 当前例子

* `skills/ll-product-raw-to-src/`
* `skills/ll-dev-tech-to-impl/`
* `skills/l3/ll-execution-loop-job-runner/`
* `skills/l3/ll-governance-failure-capture/`

### 不得混淆

以下不是 skill authority：

* `cli/lib/skill_invoker.py`
* `cli/commands/skill/command.py`
* `$CODEX_HOME/skills/<name>/`

---

## 4.3 Tool

### 定义

`Tool` 是原子执行能力。

它必须满足三个条件：

1. 有明确 request/response schema；
2. 有明确 permission scope；
3. 有明确 evidence semantics 或最小 traceability。

### 当前仓库状态

LL v0.x 已部分冻结 tool 层，但尚未把它单独做成完整 object authority。

当前正式存在的是两类 tool carrier：

### A. Tooling Tool

允许 governed skill 调用。

已冻结边界包括：

* ADR-001 中的 `tooling CLI`；
* API contract 中的 `permission_scope: tooling-cli`；
* 典型家族：`artifact`、`validate`、`evidence`、`gate create/verify`。

### B. Control Tool

只允许 loop / human gate loop / watcher / control plane 调用。

已冻结边界包括：

* ADR-001 中的 `control CLI`；
* API contract 中的 `permission_scope: control-cli`；
* 典型家族：`loop`、`job`、`gate dispatch`、`gate close-run`。

### 规则

1. 如果一个能力是原子动作，就应优先落成 tool，而不是 skill。
2. 如果一个能力开始需要双角色治理、freeze、revision、semantic review，它就不再是纯 tool，应提升为 skill / workflow。
3. tool 只负责动作，不负责长期运行单元 ownership。

### 演进路线

LL v0.x 对 Tool 的对象化路线，按下面顺序收敛：

1. 当前阶段先承认 `tool` 是一类正式语义，但 carrier 仍由 `cli/commands/*`、`cli/lib/*` 与 protocol contract 承载。
2. 当某个 tool 具备稳定的 request/response schema、permission scope、evidence obligation，并且被多个能力复用时，可以从“CLI family”提升为“tool contract pack”。
3. 未来若 tool 继续扩大成稳定、可复用、可治理的一级对象，可以收敛为 `tools/<tool-name>/` 这样的 canonical bundle，或者收敛到等价的 SSOT/formal object 层，但不能继续只停留在脚本名和调用约定。
4. 到那一步，tool 也应像 skill 一样拥有明确的 contract、lifecycle、schema 与 evidence file，而不是只保留一个函数入口。
5. 在这之前，control command 仍可作为 tool 的 operator-facing façade，但 façade 不得替代 authority。

### 当前重要说明

## **LL 当前尚未把 Tool 冻结成独立目录级 canonical bundle；当前的正式 authority 暂时由 CLI interface family + protocol contract 承载。**

因此，新增“tool”时不能只说“写个脚本”，必须同时说明：

* 它属于 tooling tool 还是 control tool；
* request/response schema 是什么；
* permission scope 是什么；
* evidence / trace 如何回写。

---

## 4.4 Agent

### 定义

`Agent` 是执行主体。

它负责：

* 承担某个角色；
* 带着限定上下文执行；
* 使用允许的工具池；
* 在权限边界内生成、审核或决策。

### 当前仓库 carrier

* `agents/executor.md`
* `agents/supervisor.md`
* human gate operator
* Claude/Codex runtime 中的 installed adapter agent shell

### 规则

1. Agent 负责“谁来做”，Workflow 负责“按什么流程做”。
2. Agent 可以执行 workflow 的一个阶段，也可以是一个 loop 中的角色，但 agent 本身不是 workflow authority。
3. Agent 的能力边界由 skill contract、tool permission scope 与 session context 共同限制。

### 不得混淆

以下不再允许互相偷换：

* “这个 workflow 让 executor agent 跑了一次” 不等于 “workflow = agent”
* “这个 skill 里有 executor/supervisor” 不等于 “skill = agent”

---

## 4.5 Workflow

### 定义

`Workflow` 是多步流程编排与状态推进定义。

它负责：

* 规定步骤关系；
* 规定输入输出层级；
* 规定何时需要 gate、review、freeze；
* 规定允许的状态迁移与下游 handoff。

当前仓库中，workflow 的 authority 主要落在：

* `ll.contract.yaml`
* `ll.lifecycle.yaml`
* workflow runtime scripts
* route key，例如 `workflow.product.src_to_epic`

### 规则

1. Workflow 是静态流程 authority，不等于一次 command，也不等于某个 agent 的一次运行。
2. Workflow 可以由一个 canonical skill bundle 承载，也可以在 skill 内部继续拆 phase。
3. Workflow 的运行实例通过 `run_ref`、job、session residue、artifacts 来体现，但这些实例不反向定义 workflow 本身。

### 当前例子

* `workflow.product.src_to_epic`
* `workflow.product.epic_to_feat`
* `workflow.dev.feat_to_tech`
* `workflow.dev.tech_to_impl`

### 不得混淆

以下都不是 workflow authority：

* `.workflow/runs/<run-id>/...` 单次残留
* `agents/executor.md` 某个角色 prompt
* `ll loop run-execution` 这种 control command

---

## 4.6 Task

### 定义

`Task` 是一级运行对象。

它必须具备：

* 稳定 identity；
* owner / claim 语义；
* status 语义；
* authoritative input refs；
* retry / resume / recovery 语义；
* evidence / outcome writeback。

### 当前仓库决定

LL v0.x 当前把 runtime task authority 暂时收敛为：

## **`job object family`**

也就是：

* `artifacts/jobs/ready/*.json`
* `artifacts/jobs/running/*.json`
* `artifacts/jobs/done/*.json`
* `artifacts/jobs/failed/*.json`
* `artifacts/jobs/waiting-human/*.json`
* `artifacts/jobs/deadletter/*.json`

`cli/lib/job_state.py`、`cli/lib/job_queue.py`、`cli/commands/job/command.py`、`cli/commands/loop/command.py` 已经提供：

* claim
* run
* renew lease
* complete
* fail
* recover
* waiting-human release

这些能力说明：

## **runtime task 在当前仓库里已经存在，只是名字暂时收敛在 job，而不是 generic task。**

### 关键边界

1. `impl-task.md` 是 Artifact，不是 runtime task。
2. `artifacts/task/` 目前只是预留目录，不是已冻结的 task authority。
3. 后续若引入 generic task object，`job` 应视为其运行时 specialization，而不是另起一套平行事实源。

### 一句话规则

> 需要 claim / owner / retry / resume / audit 的对象，先按 Task 建模；在当前仓库里，Task 先落成 Job，而不是先写一份 markdown。

---

## 4.7 Session

### 定义

`Session` 是可持续运行、可 compact、可 resume、可 handoff 的长期上下文对象。

它承载的是：

* 对话上下文；
* 阶段性状态；
* workspace residue；
* 运行中间痕迹；
* 尚未提升为正式 artifact 的过程信息。

### 当前仓库 carrier

* `.workflow/claude-code/*`
* `.workflow/workspace/*`
* `.workflow/runs/*`
* `.workflow/events.jsonl`
* `.workflow/orchestrator.db`
* `artifacts/active/runner/contexts/*.json`

### 规则

1. Session 是 runtime object，不是 SSOT artifact。
2. Session 的原始 residue 默认保留在 `.workflow/` 等 runtime 区域，不应直接冒充 formal object。
3. 只有当 session 输出被投影为正式 report / evidence / receipt / artifact 时，才进入 `artifacts/` 或 `ssot/`。

### 重要补充

本 ADR 不把 `run` 单独提升为新的一级抽象。

在 LL v0.x 中：

* `run_ref` 的 primary role 是 trace / correlation identity，用于串联 session、task、job、artifact、evidence 与 gate 事件；
* `run_ref` 不是独立 authority object，也不是调度 authority；
* `run_ref` 可视为 session 内的一次流程执行 occurrence；
* 它服务于追踪，但不替代 session、workflow 或 task 的定义。

---

## 4.8 Artifact

### 定义

`Artifact` 是正式持久化、可引用、可评审、可复用的产物对象。

它可以是：

* 业务主链对象
* 派生设计对象
* formal publication
* package
* handoff
* report
* receipt
* queue-visible runtime record

### 规则

1. Artifact 的核心责任是“成为稳定引用对象”，而不是“只是某个目录里的文件”。
2. Artifact 可以是业务对象，也可以是治理对象。
3. Task、Evidence、Gate 往往最终都以 artifact file 形式落盘，但其架构 identity 不应被“只是文件”抹平。

### 4.8.1 Artifact 子类边界

为了避免 `Artifact` 再次变成兜底词，本 ADR 进一步把它分成三类：

| 子类 | 正式含义 | 典型例子 | 主要判断 |
| --- | --- | --- | --- |
| business artifact | 业务真相或业务派生真相的正式对象 | `SRC`、`EPIC`、`FEAT`、`TECH`、`IMPL`、formal publication | 它是否主要承载产品/业务/设计真相 |
| governance artifact | 用于审批、审计、修复、回归、门禁的正式对象 | gate decision、review report、diagnosis report、regression case、evidence bundle | 它是否主要承载治理判断与证明链 |
| runtime-projected artifact | 由 session / run / task 投影出来、供后续引用或恢复的对象 | runner context、entry receipt、execution attempt record、session snapshot | 它是否主要承载过程投影而非长期真相 |

### 判定规则

1. 如果对象主要回答“系统里真正成立的事实是什么”，优先判为 business artifact。
2. 如果对象主要回答“为什么可以推进、为什么被拒绝、为什么要修”，优先判为 governance artifact。
3. 如果对象主要回答“这次运行发生了什么、如何恢复、如何串联 trace”，优先判为 runtime-projected artifact。
4. runtime-projected artifact 只有在被稳定引用、反复消费、并脱离单次 session 语境后，才有机会升级为 business artifact 或 governance artifact。

### 当前例子

* `artifacts/src/*`
* `artifacts/feat/*`
* `artifacts/tech-to-impl/*`
* `artifacts/active/handoffs/*`
* `artifacts/active/receipts/*`
* `artifacts/registry/*`
* `ssot/*`

---

## 4.9 Evidence

### 定义

`Evidence` 是支撑结论、Gate、审计、恢复或监督判断的专用 artifact。

它的责任不是“成为主输出”，而是“为某个判断提供可追溯依据”。

### 当前仓库 carrier

* `artifacts/evidence/execution/*`
* `artifacts/evidence/supervision/*`
* `artifacts/reports/*`
* audit finding bundles
* execution attempt records
* `evidence_refs`

### 规则

1. 不是所有 artifact 都是 evidence。
2. 只要某对象被用来支撑 decision、approval、repair 或 regression，它就应进入 evidence 语义。
3. evidence 必须可通过 ref 被回链，不能只存在于聊天或一次性 stdout 中。

### 与 Artifact 的关系

* Artifact 是更大的集合；
* Evidence 是带证明责任的 artifact specialization。

---

## 4.10 Gate

### 定义

`Gate` 是对状态推进进行约束的治理边界。

它负责：

* 接收 handoff / package；
* 形成 brief / pending / decision；
* 判断是否允许推进；
* 在允许推进时决定 `progression_mode`；
* materialize / dispatch / hold / close-run。

### 当前仓库 carrier

* `artifacts/active/gates/briefs/*`
* `artifacts/active/gates/pending*/*`
* `artifacts/active/gates/decisions/*`
* `artifacts/active/gates/dispatch/*`
* `cli/commands/gate/command.py`

### 规则

1. Gate 只决定“能否推进”和“以什么模式推进”。
2. Runner 只消费已经被 gate 允许推进的 ready job。
3. Skill 不得自行宣告跨治理边界推进成功。

### 不得混淆

* Gate 不是 review comment；
* Gate 不是 runner；
* Gate 不是 skill 内部一个 if/else；
* Gate 的 decision instance 不能被 session residue 或 agent 口头判断替代。

# 6. 新能力落位规则

自本 ADR 生效后，新增能力必须先回答以下问题。

## 6.1 先问 primary abstraction

1. 这是显式入口，还是能力本体？
   - 显式入口：`Command`
   - 能力本体：继续分类
2. 这是原子动作，还是多步治理能力？
   - 原子动作：`Tool`
   - 多步治理能力：`Skill / Workflow`
3. 这是执行主体，还是流程定义？
   - 执行主体：`Agent`
   - 流程定义：`Workflow`
4. 这是运行中的 work unit，还是输出说明文档？
   - 运行中的 unit：`Task`
   - 输出说明文档：`Artifact`
5. 这是过程上下文，还是正式持久化结果？
   - 过程上下文：`Session`
   - 正式结果：`Artifact / Evidence`
6. 这是状态推进判断，还是普通 review？
   - 状态推进判断：`Gate`

## 6.2 再选 carrier

落位完成后，再决定：

* 放在 `skills/`
* 放在 `cli/commands/`
* 放在 `cli/lib/`
* 落在 `.workflow/`
* 落在 `artifacts/`
* 落在 `ssot/`

而不是反过来因为“先写在哪个目录里”就决定它是什么。

---

# 7. 明确反模式

## 7.1 把 command 冒充 skill

例如只加：

* `ll skill ...`
* `ll loop ...`

却没有明确 primary abstraction 与 canonical authority。

## 7.2 把脚本冒充 tool

只因为有：

* `scripts/*.py`
* shell wrapper

就直接称其为 tool，而没有：

* schema
* permission scope
* evidence semantics

## 7.3 把 agent 冒充 workflow

只因为某个 executor 在对话里走了几步，就把那轮执行直接叫 workflow。

## 7.4 把 artifact 冒充 task

典型错误包括：

* `impl-task.md` 被当成可 claim / retry / resume 的运行单元；
* `task` 被继续理解成文档标题，而不是 job-like runtime object。

## 7.5 把 session residue 冒充正式 truth

例如把：

* `.workflow/claude-code/*/conversation.log`
* 临时 workspace 文件
* runtime cache

直接当作正式 evidence 或 artifact authority。

## 7.6 把 gate 判断下沉给 skill 或 runner

skill 和 runner 都不能自判跨治理边界推进。

---

# 8. 对当前仓库的直接约束

## 8.1 `skills/` 继续只承载 Skill authority

新增 governed capability 时，若 primary abstraction 不是 skill / workflow，就不应默认先落在 `skills/`。

## 8.2 `cli/commands/` 继续承载 Command / control carrier

新增 `cli` action 时，必须同时写清：

* 它是 command、tool 还是 control surface；
* 是否只是某个 skill 的 carrier；
* 是否有 permission scope 限制。

## 8.3 `artifacts/jobs/` 视为当前 Task authority

后续所有 queue / retry / resume / lease / recovery 相关设计，默认必须围绕 `job object family` 继续收敛，不得重新退回“靠文档暗示任务状态”。

## 8.4 `.workflow/` 继续视为 Session / runtime residue 区

除非投影为正式 artifact，否则 `.workflow/` 中的内容不应被当成长期 authority。

## 8.5 `artifacts/evidence/` 与 `evidence_refs` 继续视为 Evidence authority

后续 gate、review、repair、regression 若需要证明链，必须引用正式 evidence object，而不是依赖聊天记忆。

# 9. 与既有 ADR 的关系

## 9.1 与 ADR-001 的关系

ADR-001 已冻结：

* dual-session dual-queue；
* skill 与 CLI 的初始边界；
* tooling CLI / control CLI 区分。

ADR-038 在此基础上把对象层抽象彻底说清楚。

## 9.2 与 ADR-018 的关系

ADR-018 已经把 `job runner` 与 execution loop 具体化。

ADR-038 进一步明确：

* `job` 在当前仓库里承载 `Task authority`；
* runner 是 control plane consumer，不是 skill、tool 或 gate。

## 9.3 与 ADR-020 的关系

ADR-020 已经把 `skill / adapter / CLI carrier` 边界钉住。

ADR-038 进一步补齐：

* command
* tool
* agent
* workflow
* task
* session
* artifact
* evidence
* gate

从而避免再把 skill 作为默认兜底概念。

## 9.4 与 ADR-037 的关系

ADR-037 把 failure governance object 资产化。

ADR-038 给出更上层分类：

* `failure_case / diagnosis / remediation_plan / regression_case` 首先是 Artifact；
* 它们中的证明链部分属于 Evidence；
* 它们不应被混成 session residue 或临时 note。

# 10. 实施要求

本 ADR 生效后，后续所有 Skill / Workflow / Tool 开发，至少应显式引用本 ADR 并先完成以下分类：

1. primary abstraction 是什么；
2. secondary abstraction 是什么；
3. authority 落在哪一层；
4. carrier 选在哪个目录；
5. 它与相邻概念的非目标边界是什么。

最低落地格式建议为：

```text
Primary abstraction:
Secondary abstraction:
Authority:
Carrier:
Not equal to:
```

若无法写清这五项，则说明设计尚未完成，不应直接进入实现。

---

## 10.2 分类评审清单

在评审一个新能力是否分类正确时，至少要过下面这组问题：

1. 是否存在长期状态 ownership？
   - 有：不能只判为 Tool
   - 没有：继续看是否只是 Command 或 Artifact
2. 是否涉及阶段推进、状态迁移、gate、handoff、review 编排？
   - 有：必须显式回答 Workflow / Gate 关系
   - 没有：Skill 可能成立
3. 是否存在 claim、lease、resume、retry、recover？
   - 有：必须落到 Task / Job 语义
   - 没有：检查是否只是一次性 command 或 tool
4. 是否只是 operator-facing trigger 或 façade？
   - 是：primary 不应自动是 Skill
   - 否：继续看能力模板与流程编排谁是主语义
5. 是否只是 runtime residue、临时记录或可丢弃上下文？
   - 是：不得升成 Artifact authority
   - 否：检查是否应进入 formal artifact / evidence
6. 是否需要 schema、permission scope、traceability？
   - 需要：它至少是 Tool 候选
   - 不需要：更可能是 Command / Agent / Artifact 的 carrier
7. 是否可以脱离当前目录或脚本实现而保持定义不变？
   - 可以：说明 authority 已经写对
   - 不可以：说明你可能只是先写了 carrier
8. 是否能用一句话清楚说明“它不是哪个概念”？
   - 不能：说明边界还没有收紧

通过标准不是“看起来像”，而是“按这些问题问下来不会反复跳层”。

# 11. 最终结论

LL v0.x 后续 Runtime 与能力设计，正式以本 ADR 为统一抽象边界基线：

本 ADR 自冻结起，作为项目 runtime 对象分层与抽象分类的核心规范；相关 skill authoring、skill install、CLI/runtime SSOT 与设计评审必须显式引用本 ADR。

## **Command 是入口，Skill 是能力模板，Tool 是原子动作，Agent 是执行主体，Workflow 是流程定义，Task 是运行单元，Session 是长期上下文，Artifact 是正式产物，Evidence 是证明型产物，Gate 是状态推进门禁。**

当前仓库中的落地收敛也同时明确为：

* `skills/` = Skill authority
* `cli/commands/` = Command / control carrier
* `tooling CLI / control CLI + protocol contract` = 当前 Tool authority carrier
* `artifacts/jobs/*.json` = 当前 Task authority
* `.workflow/` = Session / runtime residue
* `artifacts/` 与 `ssot/` = Artifact authority
* `artifacts/evidence/` 与 `evidence_refs` = Evidence authority
* `artifacts/active/gates/*` = Gate authority carrier

从本 ADR 起，LL 不再接受：

* 用 skill 兜底一切对象；
* 用 agent 替代 workflow；
* 用脚本替代 tool；
* 用 markdown 文件替代 runtime task；
* 用 session residue 替代正式 artifact；
* 用 runner 或 skill 替代 gate。
